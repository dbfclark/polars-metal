# M1 — first real GPU path: scan / project / filter

**Status.** Draft, approved 2026-05-20.
**Master plan.** [`2026-05-19-master-plan-design.md`](2026-05-19-master-plan-design.md). This document fulfills M1's slot.
**Scope.** The design for milestone M1 — not its implementation plan. The implementation plan is the next artifact, produced from this spec.

## Deliverable

After M1 ships:

- `df.collect(engine=MetalEngine())` runs at least one MSL kernel on the GPU when the query matches the closed set of supported shapes (Section 1). Unsupported shapes fall back cleanly to CPU as in M0.
- `df.filter((pl.col("a") > 0) & (pl.col("b") < pl.col("c"))).select("a", "b")` on i64 / f64 / bool columns runs the comparison, AND/OR, and compaction kernels on GPU and returns a result byte-identical to CPU Polars.
- The conformance suite (Polars' own `tests/unit/lazyframe/` + filter / comparison / select / binary-expr operation tests) passes under `engine=MetalEngine()` with zero new failures vs. the M0 baseline.
- The differential harness (`tests/diff/`) covers filter / project / predicate-expression random inputs with null-heavy strategies; M0's "bare scans only" coverage gap is closed for the M1 op set.
- A criterion + pytest-benchmark perf gate runs as part of `make bench`; M1's bar is `engine=MetalEngine()` no more than 5% slower than `engine="cpu"` on M2 Ultra across five defined queries (single-sided; faster is fine, not required).

## Out of scope for M1

- Arithmetic kernels (`+`, `-`, `*`, `/`) — M2.
- Reductions (`sum`, `mean`, `min`, `max`, `count`) — M2.
- `is_null`, `NOT`, casts inside predicates — M2.
- Strings, joins, sorts, groupby, window functions — later milestones.
- Multi-chunk frames. M1 handles single-chunk frames; multi-chunk fall back. (Polars' default `collect()` produces single-chunk in nearly all cases, so this is a tight restriction in practice, not a wide one.)
- Numeric types outside i64 / f64. Bool is supported only as a column dtype and predicate result, not as an arithmetic type.
- Portability docs (`docs/portability.md`). Deferred to M2; the portability *gate* still runs at the M1 milestone boundary using the M0 ad-hoc procedure.

## Why this slice

M0 de-risked the architecture: walker shape, buffer bridge, FFI strategy, error propagation, build wiring. What's still unknown is *kernel shape* — the null-aware MSL idiom every subsequent kernel inherits — and the IR-adapter's per-node decision logic. Both are exercised end-to-end only by running real queries through them, which is why M1 takes a vertical slice (scan → project → filter) instead of building kernels in isolation.

Filter is the canonical first kernel because it forces every hard problem in one place: null-aware bit-packed data, parallel prefix-sum-and-scatter, atomic validity writes, and a non-trivial output layout. If we can do filter right, the patterns generalize.

## Scope (decisions locked)

### IR nodes handled natively on GPU

The walker recognizes three Polars IR node shapes and dispatches GPU kernels for them. All other nodes return `FallBack`, poisoning their parent subtree.

- **DataFrameScan.** The post-I/O materialized frame. The file reader stays on CPU; we intercept the resulting in-memory DataFrame and wrap its i64 / f64 / bool Arrow buffers as `MetalBuffer`s. Zero-copy where M0's bridge allows; copy otherwise.
- **SimpleProjection / Select (metadata-only).** Column re-selection / re-ordering. Pure metadata on our side: a new column list pointing at the same `MetalBuffer`s.
- **Filter.** Predicate walked at plan time against a closed set (below). If every predicate node is accepted, the filter runs on GPU. If any node is rejected, the whole filter falls back — no partial dispatch within a predicate.

### Predicate expression shapes accepted

The walker accepts a `Filter` node iff every node in the predicate expression is one of:

- `Column(c)` where `c` is i64 / f64 / Boolean
- `Literal(scalar)` of i64 / f64 / Boolean
- `BinaryExpr(==, !=, <, <=, >, >=)` where both sides resolve to a numeric leaf (column or literal) of the same type
- `BinaryExpr(&, |)` where both sides resolve to Boolean (column, literal, or accepted comparison)

Anything else (arithmetic in the predicate, `is_null`, `NOT`, casts, multi-column expressions, string ops) means the whole filter falls back.

### Type matrix

i64 + f64 + Boolean only. Other dtypes fall back at the scan handler — the column is left as a CPU column in the result frame, and any node consuming it falls back automatically. M2 expands the matrix when it expands the operator set.

### Kernels delivered

Five `.metal` files in `shaders/`, with the following entry points:

| File | Entry points |
|---|---|
| `cmp_i64.metal` | `cmp_i64_{eq,ne,lt,le,gt,ge}` and `cmp_i64_{eq,ne,lt,le,gt,ge}_scalar` |
| `cmp_f64.metal` | mirror of cmp_i64, with Polars NaN semantics |
| `logical_bool.metal` | `bool_and`, `bool_or` (3-valued) |
| `filter_predicate.metal` | `filter_predicate_to_u8` (bit-packed bool + validity → dense u8) |
| `filter_scatter.metal` | `filter_scatter_{i64,f64,bool}` |

### MLX usage

One op: `cumsum` over the `u8` array from `filter_predicate.metal`. Wrapped in `polars-metal-mlx-sys` as `cumsum_u8_to_u32`, forced onto `Device::gpu` via the same `StreamContext` pattern as M0's GPU dispatch test. No other MLX in M1.

## Architecture

### How M1 extends M0's walker

M0's `execute_with_metal` walked the IR via `nt.view_current_node()`, saw every node, and returned without calling `nt.set_udf` — Polars' CPU executor took over.

M1's structural change is:

1. The walker recurses bottom-up, producing for each IR node a `Handled(MetalPlanNode)` or `FallBack` value.
2. `Handled` carries enough metadata to rebuild the subtree as a UDF that materializes a `polars.DataFrame` from `MetalBuffer`-backed columns.
3. The walker collapses contiguous `Handled` regions into the largest GPU subtree it can. Any `FallBack` poisons its parent — same per-node-must-agree model as cuDF-polars.
4. When the whole root is `Handled`, the walker calls `nt.set_udf(metal_udf)` and Polars hands off. When the root is partly `Handled`, no `set_udf` call is made; the whole query falls back. (M1 does not lift partial subtrees to GPU — see "Partial dispatch policy" below.)

### MetalPlanNode intermediate IR

A small Rust enum, one variant per accepted node type (Scan, Project, Filter). Its purpose is to isolate "did we accept this Polars node" from "how do we dispatch kernels for it." That separation keeps the kernel crate free of Polars-IR dependencies — kernels operate on plain `MetalBuffer` references plus a small predicate AST, not on Polars node IDs.

### Shader compilation

All five `.metal` files compile to a single `polars_metal.metallib` at build time via `build.rs` in `polars-metal-kernels` (invokes `xcrun metal` + `xcrun metallib`). The metallib is embedded in the dylib via `include_bytes!`, parsed once at first use (`OnceLock`), and pipeline-state-objects are cached per kernel entry point. Threadgroup size queried from `MTLDevice.maxThreadsPerThreadgroup()` at runtime, then bounded by each kernel's preferred size — never hardcoded (per CLAUDE.md's portability gotcha).

### Arena usage

M0's `StubArena` becomes a real `ScratchArena` in M1. The arena owns:

- Predicate intermediate buffers (one per non-leaf predicate AST node — comparison outputs, AND/OR outputs)
- The `keep_flags: u8[n_rows]` buffer fed to MLX cumsum
- The `prefix_sum: u32[n_rows]` buffer from MLX cumsum
- Output column data + validity buffers for the result DataFrame

All allocated in `MTLResourceStorageModeShared`. Unified memory means CPU and GPU address the same bytes — no host↔device blits anywhere in the pipeline.

### Buffer ownership across the UDF boundary

Output `MetalBuffer`s have to outlive the UDF call because Polars' Arrow buffers (built from them) live in the returned DataFrame. The lifetime model: the arena is held in an `Arc`, and a clone is captured in each output Arrow buffer's custom-deallocator closure (mirror of M0's zero-copy bridge — there the closure keeps an Arrow buffer alive for Metal; here it keeps a `MetalBuffer` alive for Polars). When all output Arrow buffers drop, the arena drops, freeing all `MTLBuffer`s.

### MLX dispatch

`polars-metal-mlx-sys` gets one new export:

```
cumsum_u8_to_u32(input: *const u8, len: usize, out: *mut u32) -> Result<(), MlxError>
```

Wraps `mlx::core::cumsum` on the u8 view, with the output cast to u32. Forced `Device::gpu` via `StreamContext`.

## Data flow

Walk through `df.filter((pl.col("a") > 0) & (pl.col("b") < pl.col("c"))).select("a", "b")` where `a`, `b`, `c` are all i64 with nulls.

1. **Python entry.** `df.collect(engine=MetalEngine())` hits the patched `LazyFrame.collect`. We inject `execute_with_metal` as `post_opt_callback`, call original `collect(engine="cpu", ...)`.
2. **Polars optimizer runs, callback fires.** The IR (post-optimization):
   ```
   SimpleProjection(["a", "b"])
     └─ Filter(predicate = BinaryExpr(&, gt_expr, lt_expr))
           └─ DataFrameScan(a, b, c)
   ```
3. **Bottom-up walk.** `DataFrameScan` → all columns i64, returns `Handled(Scan { device_columns })`. `Filter` → predicate validator accepts all leaves, returns `Handled(Filter { input, predicate_ast })`. `SimpleProjection` → returns `Handled(Project { input, columns })`. Walker collapses to one `Handled` subtree and calls `nt.set_udf(metal_udf)`.
4. **UDF execution.**
   - Materialize scan: wrap i64 buffers via the buffer bridge (`c` included — filter needs it even though the projection drops it).
   - Allocate arena sized to roughly `5 × n_rows × 8 bytes` upper bound.
   - Evaluate predicate AST:
     - `cmp_i64_gt_scalar(a, 0)` → `tmp_gt`
     - `cmp_i64_lt(b, c)` → `tmp_lt`
     - `bool_and(tmp_gt, tmp_lt)` → `pred`
   - Compaction:
     - `filter_predicate_to_u8(pred)` → `keep_flags: u8[n_rows]`
     - MLX `cumsum_u8_to_u32(keep_flags)` → `prefix_sum: u32[n_rows]`
     - Read `prefix_sum[n_rows - 1]` from shared memory (no blit) → `n_out`
     - Allocate `a_out`, `b_out` from arena (size `n_out`)
     - `filter_scatter_i64(a, keep, prefix_sum, a_out)`, same for `b`. `c` is dropped.
   - Project: build result column list `[a_out, b_out]`.
   - Assemble Polars `DataFrame`; Arrow buffers wrap `MetalBuffer`s with arena-keep-alive deallocators.
5. **Polars sees a DataFrame.** Query done.

### Partial dispatch policy

M1 holds the line: GPU subtree must reach the root or the whole query is CPU. If the filter predicate has `is_null` (not accepted), the filter returns `FallBack`, poisoning the projection above it. No `set_udf` call. CPU runs everything.

Rationale: lifting partial subtrees to GPU means materializing scan output onto GPU, blitting back to CPU, then running CPU filter — guaranteed slower than the all-CPU path. M2's elementwise + reductions will fill in enough of the predicate set that fallback rates drop sharply; partial-dispatch heuristics are post-M3.

## Kernel inventory

Each kernel reads data + validity (bit-packed little-endian Arrow) and writes both. M0's `set_valid` / `get_valid` / `validity_bytes` helpers are ported to MSL — the same bit fiddling, in MSL. The MSL helpers live in a small `shaders/_validity.metal` header included by every kernel; not a separate entry point, just shared device functions.

### Comparison kernels (`cmp_i64.metal`, `cmp_f64.metal`)

Six entry points each (eq, ne, lt, le, gt, ge), plus six `_scalar` variants that take a 1-element rhs (avoids materializing a length-N literal column). Signature pattern:

```msl
kernel void cmp_i64_lt(
    device const int64_t* lhs_data       [[buffer(0)]],
    device const uint8_t* lhs_valid      [[buffer(1)]],
    device const int64_t* rhs_data       [[buffer(2)]],
    device const uint8_t* rhs_valid      [[buffer(3)]],
    device       uint8_t* out_data       [[buffer(4)]],   // bit-packed bool
    device       uint8_t* out_valid      [[buffer(5)]],   // bit-packed
    constant     uint32_t& n_rows        [[buffer(6)]],
    uint                  gid            [[thread_position_in_grid]]);
```

**Null rule.** `out_valid[i] = lhs_valid[i] AND rhs_valid[i]`. If either input is null, output is null. Per-row data is written unconditionally — null slots have unspecified bits (Polars convention).

**f64 NaN rule (Polars-conformant).**
- `NaN < x`, `NaN <= x`, `NaN > x`, `NaN >= x`, `NaN == x` → `false`
- `NaN != x` → `true`
- NaN's validity bit is still set (NaN is a value, distinct from null)

### Logical kernels (`logical_bool.metal`)

Two entry points: `bool_and`, `bool_or`. Inputs and outputs are bit-packed bool columns + validity bitmaps. Implemented bitwise on `u8` blocks of 8 rows.

**3-valued AND truth table:**

| lhs | rhs | result |
|---|---|---|
| true | true | true |
| true | false | false |
| true | null | null |
| false | true | false |
| false | false | false |
| false | null | false |
| null | true | null |
| null | false | false |
| null | null | null |

(OR is symmetric.) The truth table is asserted exhaustively in the unit test; the implementation works in 8-row blocks via bit operations on `(data, validity)` pairs.

### Predicate compaction kernels

**`filter_predicate.metal::filter_predicate_to_u8`.** Reads a bit-packed bool column + its validity, writes a dense `u8[n_rows]` where `out[i] = 1` iff `data[i] AND valid[i]`. 8× the bytes of the bit-packed form; that's the price for MLX cumsum's dense-input requirement.

**`filter_scatter.metal::filter_scatter_{i64,f64,bool}`.** Reads source column + validity, the `keep: u8[n_rows]` flags, and `prefix_sum: u32[n_rows]` (inclusive scan). Writes surviving rows to `dst_data[prefix_sum[gid] - 1]` and ORs in the validity bit.

**Validity-bit atomics.** Eight output rows share a `u8` in `dst_valid`. The scatter zero-inits the validity buffer first, then each thread writing a valid row uses `atomic_fetch_or_explicit` on a `device atomic_uint*` view of the validity buffer (validity is round-up-allocated to 4-byte alignment for atomic compatibility). For i64 and f64 scatters, data writes are race-free because each `out_idx` is unique to one thread.

**Bool data scatter — atomics for data too.** `filter_scatter_bool` is the only variant where the *data* output is also bit-packed, so it inherits the same multi-thread-same-byte problem as validity. Same fix: atomic OR on the data buffer (when source bit is 1) using a `device atomic_uint*` view, with the data buffer zero-inited first.

**Scatter overrun sentinel (release-build safety).** Before allocating the output column, we read `prefix_sum[n_rows - 1]` as `n_out`. We allocate `n_out + 1` slots and the scatter writes a sentinel to slot `n_out` if it ever computes `out_idx >= n_out`. CPU checks the sentinel post-dispatch and raises `EngineError::ScatterOverrun` if tripped. Cheap; catches the class of bugs that would otherwise silently corrupt arena memory.

## Error & fallback policy

Two categories with different handling:

**Category A — plan-shape rejections (expected, frequent).** The walker meets a node or expression shape it can't handle. Returns `FallBack`; walker does not call `set_udf` on the poisoned subtree; Polars runs CPU. Not logged at default level; logged at DEBUG when `MetalEngine(debug=True)`.

**Category B — runtime failures during GPU execution (rare, fatal).** Once `set_udf` is called, the UDF must produce a correct result or fail loudly. **No mid-execution fallback.** Failure modes:

- MTLBuffer allocation failure (unified memory exhausted) → `EngineError::ArenaExhausted`
- MLX cumsum failure → `EngineError::Ffi`
- Kernel dispatch failure (`MTLCommandBuffer` errors after `waitUntilCompleted`) → `EngineError::KernelDispatch`
- Scatter overrun sentinel tripped → `EngineError::ScatterOverrun`

All convert to `polars.exceptions.ComputeError` at the PyO3 boundary (pre-M1 fix, commit `c6d9558`).

**Why no mid-execution fallback.** Mid-execution fallback would mean: (a) arena state already mutated, partial dispatches issued — work wasted; (b) walker's accept/reject decision already made — falling back at execution time relitigates a plan-time decision; (c) silent fallback masks real bugs. A kernel that should produce results but silently falls back would never get a bug report.

**Walk-time validation is strict.** The predicate validator in `walker/filter.rs` is the gatekeeper. If it accepts a shape the kernels can't actually handle, that's a category B failure at runtime. Tightening the validator is the fix, not loosening the runtime.

**Debug-build invariant checks.** In `cargo build` (debug only), each kernel dispatch is followed by CPU-side checks on sampled outputs: validity bits in range, scatter output indices unique, prefix-sum monotonic. Stripped in `--release`. These are `polars-metal-kernels`'s responsibility, not the walker's.

**Logging.** `MetalEngine(debug=True)` enables walker decisions ("accepted Filter" / "fell back Filter — predicate has is_null"), per-kernel dispatch timing, and scan-path alignment regime taken. Routed through Python `logging.getLogger("polars_metal")`. Never `println!` from Rust.

## Testing strategy

Four layers. Each catches a specific failure mode.

### Layer 1 — Kernel-level correctness (`tests/kernel/`)

One Rust integration test file per `.metal` file. Each test builds a `MetalDevice`, allocates input/output buffers, dispatches the kernel, asserts byte-exact output against a pure-Rust reference.

Proptest coverage per kernel (256 cases per property, reproducible failure seeds checked in):

- **`cmp_i64`** — random i64 columns + validity bitmaps, every density 0–100% null, every op, both column-column and column-scalar variants.
- **`cmp_f64`** — same plus NaN-injection strategy (~30% NaN, ±Inf, subnormals).
- **`logical_bool`** — exhaustive truth table (9 input combinations) plus proptest for 8-row-at-a-time bit-packing.
- **`filter_predicate`** — random bit-packed bool + validity; assert dense u8 output equals `(data[i] & valid[i])`.
- **`filter_scatter`** — random source + keep patterns; output equals "source rows where keep is true"; validity-write atomic correctness with `keep` patterns that force multi-thread same-byte writes.

### Layer 2 — Differential testing (`tests/diff/`)

Hypothesis-based Python. Random small DataFrames (1–10K rows, 1–4 columns, supported dtypes, 0–100% null), random M1-shape queries, asserts `assert_frame_equal(cpu_result, gpu_result)` byte-exact.

Three new strategy generators (M0's harness only covered bare scans, per the M0 retrospective):

- `m1_filter_strategy` — predicates from the closed set in Section 1, recursive up to depth 3.
- `m1_projection_strategy` — column subsets in random orders.
- `m1_null_density_strategy` — weights null density toward 0% and 100% (most likely to expose bit-packing bugs).

Explicit edge cases as named pytest functions in `tests/diff/test_filter_edges.py`:
- Empty DataFrame (0 rows, schema present)
- Single-row DataFrame, predicate true / false / null
- All-null column
- All-NaN f64 column
- Predicate produces all-null result
- Predicate produces all-false result (output 0 rows, schema preserved)
- Predicate produces all-true result (output identical to input)

### Layer 3 — Conformance (Polars' own test suite, `tests/conformance/`)

M0 wired Polars' `tests/unit/lazyframe/` under `engine=MetalEngine()` (721 pass / 0 fail / 1 skip). M1 extends to:

- `tests/unit/operations/test_filter.py`
- `tests/unit/operations/test_comparison.py` (or equivalent — pin to the file in Polars `py-1.40.1`)
- `tests/unit/operations/test_select.py`
- `tests/unit/expr/test_binary.py`

**Bar.** Pass count under `engine=MetalEngine()` equals pass count under CPU. Any new failure is a release-blocker. Skip/xfail list in `tests/conformance/skip_registry.py`; adds require a PR comment explaining why and a tracking issue.

**M1 skip allowances.** Tests exercising dtypes outside i64/f64/bool or operations outside the M1 set may be skipped. A test touching only M1-supported shapes that fails under our engine is a bug.

### Layer 4 — Benchmarks (`tests/bench/`, `benches/`)

**Criterion (Rust, per-kernel).** One bench group per `.metal` file. Pure kernel time, excluding IR walk / Polars overhead. Sizes: 1K, 100K, 10M rows. Null densities: 0%, 50%, 100%. Reports in `target/criterion/`.

**pytest-benchmark (end-to-end).** Five queries:

- `bench_filter_simple` — `df.filter(pl.col("a") > 0)` on 10M-row i64, 50% null density
- `bench_filter_compound` — `df.filter((pl.col("a") > 0) & (pl.col("b") < pl.col("c")))`
- `bench_filter_then_project` — same as compound + `.select("a", "b")`
- `bench_filter_then_project_high_selectivity` — predicate matches 1% of rows
- `bench_filter_then_project_low_selectivity` — predicate matches 99% of rows

**Bar.** `engine=MetalEngine()` is not slower than `engine="cpu"` on M2 Ultra by more than 5% on any of these queries at any size class. Baseline numbers in `tests/bench/baseline.json`; regressions show in PR diffs. The ±5% band may be widened if measurement noise on M2 Ultra makes it impractical — to be re-evaluated against real numbers, not in advance.

### Layer 5 — Portability gate (manual, milestone boundary)

`make gate` run on M2 Ultra (primary), M2 (16 GB), M1 (8 GB). Uses the M0 ad-hoc procedure; documentation deferred to M2. Catches threadgroup-size mistakes and OOM regressions on the 8 GB box.

## Exit criteria

M1 is done when all of the following are simultaneously true.

**Functional.**

1. `df.collect(engine=MetalEngine())` on any query matching the closed set runs at least one MSL kernel on GPU. Verified by `MetalEngine(debug=True)` log assertions in the conformance suite.
2. Any unsupported shape falls back cleanly to CPU. No partial GPU work, no errors.
3. `polars.exceptions.ComputeError` is the exception type for any category-B failure. Asserted by a dedicated test.

**Correctness.**

4. `make test-kernel` passes: every kernel proptest green at 256 cases, with NaN/edge cases.
5. `make test-diff` passes: hypothesis filter/project/predicate strategies green; all explicit edge cases pass.
6. `make test-conformance` passes with zero new failures vs. M0's baseline (721/0/1). The M1 skip registry has zero entries for M1-supported shapes.

**Performance.**

7. `make bench` runs with no engine=MetalEngine() result >5% slower than engine="cpu" on M2 Ultra across all five queries and all size classes. Baseline numbers checked into `tests/bench/baseline.json`.

**Quality.**

8. `make gate` passes locally on M2 Ultra.
9. Portability gate passes on M2 (16 GB) and M1 (8 GB); results (date + git SHA + summary) recorded in the M1 retrospective.
10. `cargo clippy --workspace --all-targets -- -D warnings`, `cargo fmt --check`, `ruff check python/`, `ruff format --check python/` all clean.

**Documentation.**

11. `docs/architecture.md` extended with the M1 walker + plan IR + kernel dispatch picture.
12. `docs/kernel-authoring.md` extended with the null-aware MSL idiom — comparison kernels are the canonical example.
13. `docs/open-questions.md` updated: M1-owned items resolved or moved; new M1-surfaced questions added with owners.
14. M1 retrospective written in this spec's closing section, following M0's template.

**Hand-off to M2.**

15. M2's null-aware reductions strategy left genuinely open in the M1 retrospective. M1's comparison kernels establish a precedent for null-aware MSL but don't commit M2's reductions choice.
16. The MLX FFI choice gets its scheduled M2 revisit. M1 surfaces one new MLX op (cumsum); the friction or lack thereof feeds the M2 reconsideration.

## Risks & open questions (M1-owned)

**Filter compaction allocation overhead.** The `keep_flags: u8[n_rows]` buffer is an 8× blowup over bit-packed bool, accepted in exchange for MLX cumsum. If profiling shows this hurts (large frames on memory-constrained machines), M2 or later may revisit either a hand-rolled bit-packed scan or a fused predicate-AST kernel. Tracked here, owned by post-M1 perf analysis.

**Scatter atomics on bool data.** Validity atomics are uniform across scatter types, but `filter_scatter_bool` adds a second atomic on the bit-packed *data* buffer. Risk: a kernel-level bug that only fires on specific keep-flag distributions (e.g., keep patterns that put exactly one true row in every 8-row block). Mitigation: proptest with keep patterns explicitly engineered to force every output byte to be touched by ≥2 threads for both data and validity.

**Predicate validator drift.** Walker-side validator and runtime kernels must agree on what's supported. If a predicate shape is accepted that no kernel handles, that's a category B runtime failure. Mitigation: validator + dispatch share a single Rust enum (the accepted-leaf-and-binary-op set) — adding a kernel without adding it to the enum, or vice versa, fails to compile.

**Polars rev drift.** M0 pinned Polars to 1.40.1. M1 doesn't bump. If a Polars patch release lands during M1 with a relevant fix, the decision is "stay pinned and ship M1" — bumping happens at milestone boundaries per the master plan.
