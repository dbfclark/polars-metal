# M3 — Real-workload quality: canonical TPC-H slice, dual-mode GPU build, multi-agg fusion, strings — Design

**Status:** approved 2026-05-22, ready for implementation plan.

**Predecessor:** M2 (`docs/superpowers/specs/2026-05-21-m2-design.md`) landed the per-op routing layer and the M2 modified TPC-H Q1 at ratio 0.914. The M2 perf finding (`docs/open-questions.md` § "M2 perf finding") established that GroupBy aggregation kernels are **not** the bottleneck; build phase, fusion, and filter-into-GPU routing have measurable headroom. M3 takes that finding and uses a TPC-H slice as the forcing function for capability scope.

## Mission

Ship `polars-metal` at real-workload quality on the analytical surface M2 staked out. Land four TPC-H-shaped queries — canonical Q1, high-cardinality Q1 variant, Q6, 100M-row Q1 — with each strictly faster than CPU Polars on M2 Ultra. The capabilities required (GPU build phase dual-mode, multi-aggregation kernel fusion, filter-into-GPU routing, string-key groupby, multi-chunk Series, smaller-integer key dtypes, expression unfolding) are what M3 ships.

Reframed from M2: M2 proved "the engine can beat CPU on a curated query." M3 proves "the engine can beat CPU on real-shaped queries — not just curated."

**Perf gate tightens from `< 1.0` (M2) to:**

- `ratio < 0.7` on canonical Q1, high-card-64K, Q6
- `ratio < 1.0` on high-card-1M
- `ratio < 0.5` on 100M Q1 (overhead-dominance fades at scale)

## Scope (decisions locked)

### Workload validated

Four query shapes, five benchmark entries (high-card split into 64K and 1M to exercise the A1↔A2 boundary). Sort step routes to CPU on all (no GPU sort op shipped in M3).

| Entry | Shape | Drives capability | Target ratio |
|---|---|---|---|
| **Canonical Q1** | filter → groupby[Utf8×2] → 8 aggs incl. `sum(a*b)` → sort | D (strings), G (expression unfolding), B (8-agg fusion) | `< 0.7` |
| **High-card Q1, 64K groups** | filter → groupby[int, 64K groups] → 5 aggs → sort | A1 (partitioned hash), B | `< 0.7` |
| **High-card Q1, 1M groups** | filter → groupby[int, 1M groups] → 5 aggs → sort | A2 (sort-segment-reduce), B | `< 1.0` |
| **Q6** | filter (multi-predicate) → single-group aggregation (`sum(l_extendedprice * l_discount)`) | C (filter→GPU when downstream-GPU), B, G | `< 0.7` |
| **100M Q1** | M2's modified Q1 at 10× scale | scale validation | `< 0.5` |

All fixtures deterministic, in `tests/bench/`. Canonical Q1 fixture adds `l_returnflag`/`l_linestatus` as Polars `Utf8` (not int-encoded as M2). High-card variant adds synthetic `customer_key` column at 64K and 1M cardinality. 100M variant is a 10× scaled `_lineitem_fixture.py`.

### IR nodes handled natively on GPU

Carries M2's set (DataFrameScan, SimpleProjection/Select, Filter, GroupBy) and extends:

- **Filter** — routing rule changes: routes to GPU when its consumer in the IR is a GPU-routed GroupBy or a GPU-routed single-group aggregation. Else CPU (M2 behavior).
- **GroupBy** — now handles Utf8 key columns (via D), expression-shaped aggregations (via G), smaller-integer composite keys (via F). The build phase dispatches dual-mode A1/A2 by cardinality estimate (via A). Aggregation pass uses one fused kernel per query signature (via B).
- **Single-group aggregation** — Q6's `df.filter(...).select(agg_expr.sum())` shape. The exact Polars IR node (likely `Select` with an `Agg` expression, possibly a distinct `Reduce` node) is verified at implementation time against the pinned `py-1.40.1` tree. The walker maps it to a `MetalPlanNode::GroupBy` with an empty key list and routes through the same build-phase-skip + fused-aggregation path (build is trivial: all rows → group 0).

All other IR node types fall back, same as M2.

### Routing decisions (cost model)

| Op | Decision | Notes |
|---|---|---|
| Filter (isolated) | CPU | M2 rule unchanged |
| Filter → GPU GroupBy | GPU (fused with consumer) | new in M3 |
| Filter → GPU single-group aggregation | GPU (fused with consumer) | new in M3 |
| Project / SimpleProjection | follow input | M2 rule unchanged |
| Scan | follow output | M2 rule unchanged |
| GroupBy | GPU iff n_rows > 100K | M2 threshold; same |
| Single-group aggregation | GPU iff n_rows > 100K | same threshold as GroupBy |
| GroupBy build mode | A1 if est_card ≤ 64K, A2 otherwise | new in M3 |

Affinity smoothing (M2) extended to recognize filter+groupby and filter+reduction as fused subtrees when both lift GPU.

### Type matrix

- **Group keys:** fixed-width (i8/i16/i32/i64, u8/u16/u32/u64, f32/f64, bool — M2 extended by F) and Utf8 (new in M3 via D). Composite encoding still capped at 128 bits total.
- **Aggregation value columns:** all M2 dtypes (i32/i64/f32/f64 native; smaller integers via cast). M2's 32-bit GPU + 64-bit CPU-finalize hybrid is preserved.
- **Expression operands in aggregations (G):** column×column or column×literal with binary ops `*`, `+`, `-`, `/`. Anything else falls back at plan time.

### Aggregations delivered

Same six entry points as M2 (sum, mean, count, min, max, `pl.len()`), now also accepting expression-shaped inputs and dispatched through the fused kernel template (B).

### MLX usage

No new MLX ops in M3. The existing `cumsum_u8_to_u32` binding is unchanged.

**Module ownership boundary.** The MSL template engine in `aggregate_fused.rs` is the most novel piece. It owns: parsing the per-query agg signature, emitting MSL source as a string, compiling via `MTLDevice::newLibraryWithSource_options_error_` at first use, caching by signature hash. It does **not** own the per-kernel dispatch loop (that stays in `groupby.rs`) or buffer layout (that stays in `polars-metal-buffer`).

The cardinality estimator (`groupby_cardinality.rs`) is the riskiest dependency: A1/A2 selection depends on it being approximately right. If estimation is wrong by 10×, the wrong build phase is picked. Mitigation: A1 detects TGSM overflow at execution time and bails out to A2 (one re-dispatch, not a query failure). A2 has no fast-path equivalent — it just runs slightly suboptimally when cardinality is low.

## Algorithm details

### A1: Partitioned hash build phase

```
Input:  encoded_keys: MetalBuffer<u128>[n_rows]
        n_partitions: u32  (computed from cardinality estimate)

Per row r:
  partition_id = (hash(encoded_keys[r]) >> log2(TGSM_slots)) & (n_partitions - 1)
  emit (r, partition_id)

Phase A1.1 — partition scatter (one MSL kernel):
  Atomic-increment per-partition row counts, scatter rows to partition
  lanes.
  Output: row_indices_by_partition: u32[n_rows],
          partition_offsets:        u32[n_partitions+1]

Phase A1.2 — per-partition hash build (one MSL kernel, n_partitions
threadgroups):
  Each threadgroup loads its slice of row_indices_by_partition.
  Builds a hash table in TGSM: open addressing, linear probe.
  Per-partition local group_ids assigned 0..k_p.
  Output: row_to_local_group:      u32[n_rows],
          n_groups_per_partition:  u32[n_partitions]

Phase A1.3 — global group_id assignment (CPU, microseconds):
  Exclusive scan over n_groups_per_partition. Add to local group_ids
  to produce global row_to_group.

Phase A1.4 — first_row_per_group: u32[n_groups_total]
  CPU pass over row_to_group (cheap).

TGSM overflow detection: if any threadgroup hits >90% load factor, it
sets a global "overflow" flag (u32 atomic). After A1.2, CPU checks the
flag; if set, re-run as A2.
```

Dimensions: 32 KB TGSM ÷ 24 bytes/slot ≈ 1365 slots/tg. At 75% load factor: ~1024 unique groups/tg. With 64 partitions: ~65K total — the A1↔A2 threshold.

### A2: Sort-then-segment-reduce build phase

```
Input:  encoded_keys:     MetalBuffer<u128>[n_rows]
        original_indices: MetalBuffer<u32>[n_rows]  (just 0..n_rows)

Phase A2.1 — GPU radix sort on (encoded_keys, original_indices):
  4 passes (32-bit lanes), each pass:
    - per-row: extract 8-bit digit from current lane
    - per-tg: build histogram in TGSM
    - global scan to find write positions
    - scatter rows by digit
  Two passes per lane (count, scatter) → 8 kernels per 32-bit lane;
  4 lanes → 32 kernels total for full u128 sort.

Phase A2.2 — segment boundaries (one MSL kernel):
  Per row r > 0: if sorted_keys[r] != sorted_keys[r-1], emit segment
  start.
  Scan to produce segment_starts: u32[n_groups+1].

Phase A2.3 — row_to_group derivation:
  Per row r: binary search segment_starts to find group_id.
  (Or: invert sorted_indices into row_to_group via scatter pass.)

Output:
  row_to_group:        u32[n_rows]    (in original row order)
  first_row_per_group: u32[n_groups]  (= original_indices at each
                                       segment_start)
  n_groups:            u32
```

Sort kernel is a **restricted-scope internal helper** for groupby only — not exposed as a Polars Sort IR node handler. M4's full radix sort milestone may generalize this kernel or keep it as a specialized fast path; that's M4's call.

Crossover with A1: at 10K groups, A1 dominates (A1: ~10 ms; A2: ~80 ms sort + 5 ms reduce). At 1M groups, A2 dominates (A1: TGSM overflows on every partition, fallback to A2; A2: still ~85 ms).

### B: Fused multi-aggregation kernel template

MSL template, parameterized by `{aggs}` list, emitted at plan time:

```msl
kernel void aggregate_fused(
  device const uint*  row_to_group [[buffer(0)]],
  device const uint*  group_count  [[buffer(1)]],
  // {{value_buffer_bindings}}
  // {{output_buffer_bindings}}
  uint gid [[thread_position_in_grid]],
  uint nrows [[grid_size]])
{
  if (gid >= nrows) return;
  uint g = row_to_group[gid];

  // {{per_agg_body}} — one stanza per aggregation:
  //   - load value (with null check)
  //   - apply op-specific atomic update on outputs[op_idx][g]
  // ALL aggs over the same value column share the load.
}
```

Rust code generation:

```rust
fn emit_msl_for(aggs: &[AggSpec], key_count: usize) -> String {
    // Group aggs by value column → emit one load per group, then
    // one update per agg in that group. Null handling per op.
    // Cache by hash of (aggs signature) so identical queries reuse.
}
```

For 32-bit hot path: kernel runs to completion on GPU. For 64-bit accumulators: kernel computes partial 32-bit segments, CPU finalizes (M2's pattern, kept).

### G: Expression unfolding inside fused kernel

`AggSpec::Expression { expr, op }` — the rewriter walks the expression tree and emits MSL inline:

```
sum(l_extendedprice * (1 - l_discount))
→
// Per row r:
float ep = value_extendedprice[r];
float disc = value_discount[r];
if (validity_extendedprice[r/8] & (1<<(r%8)) && ...) {
    float v = ep * (1.0f - disc);
    atomic_fetch_add_explicit(&output_sum[g], v, ...);  // 32-bit path
}
```

Scope: binary ops `*` `+` `-` `/` on column/column or column/literal. Anything else → fallback at plan time.

### D Phase 1: Dictionary encoding

```
Input:  Polars Utf8 series (offsets: u32[n+1], data: u8[])

CPU phase (using polars-arrow):
  dict:  Vec<&str>  = unique string values
  codes: Vec<u32>[n] = index into dict
  -> wrap codes as MetalBuffer<u32>, feed to composite-key encoder
     as a u32 key column with width 32 bits

After groupby:
  first_row_per_group → first_row's u32 code → dict[code] → result
  Utf8 column
```

Dictionary cap: 2^32 unique strings (won't hit in practice). For Q1's 2 unique values, dictionary build is in microseconds.

### D Phase 2: Dedicated string-hash kernel

```msl
kernel void hash_strings(
  device const uint*  offsets [[buffer(0)]],
  device const uchar* data    [[buffer(1)]],
  device uint*        hashes  [[buffer(2)]],
  uint r [[thread_position_in_grid]])
{
  uint start = offsets[r], end = offsets[r+1];
  uint h = 2166136261u;  // FNV-1a seed
  for (uint i = start; i < end; ++i) {
    h ^= data[i];
    h *= 16777619u;
  }
  hashes[r] = h;
}
```

Used when cardinality estimator suggests dictionary build cost > direct-hash cost. The hash becomes the u32 key component; downstream is the same partitioned-hash or sort-segment-reduce build. Hash collisions are resolved by full-string comparison in the partitioned-hash A1 inner loop.

### E: Multi-chunk Series support

Two regimes, picked at scan time by total-bytes threshold:

- **Rechunk-on-import** when total bytes < 256 MB. `series.rechunk()` produces a single-chunk view; existing kernels run unchanged.
- **Chunked dispatch** when total bytes ≥ 256 MB. Kernels accept `&[MetalBuffer<T>]` slices instead of single buffers; each kernel pass iterates chunks, accumulating into one output buffer.

Threshold tuned in microbenches. Real chunked LazyFrames (parquet row-group chunked) tested in both regimes.

### F: Smaller integer key dtypes

`KeyDtype` enum extended with `I8`, `I16`, `I32`, `U8`, `U16`, `U32`. Encoder packs each per its byte-width into the composite key. Composite-key width check still enforces ≤ 128 bits.

### C: Filter routing into GPU when downstream is GPU

Cost-model extension. When `Filter`'s consumer is `GroupBy` *and* groupby routes GPU (or is a single-group reduction routing GPU), the filter routes GPU too. The combined subtree lifts as one GPU operation; no intermediate Polars DataFrame is materialized.

The win is **skipping the materialize step**: Arrow buffer assembly, validity reconciliation, optimizer round-trip. Filter alone on GPU does not beat CPU — the unified-memory finding holds. The unfused case (filter alone) still routes CPU.

## Data flow for canonical Q1

```
[Polars IR] ─────────────────────────────────────────────────
   DataFrameScan
   ↓
   Filter (l_shipdate <= threshold)
   ↓
   GroupBy (keys=[l_returnflag (Utf8), l_linestatus (Utf8)],
             aggs=[8 specs incl. sum(a*b)])
   ↓
   Sort (key cols)
[Walker translates IR → MetalPlanNode tree]
                ↓
[Rust router computes LiftingPlan]
  Scan:    CpuLeave (inherits parent)
  Filter:  GpuLift (fused with GroupBy, new in M3)
  GroupBy: GpuLift  (10M rows > 100K; ~2 groups est_card → A1)
  Sort:    CpuLeave (sort not in M3)
[Walker applies lifting plan]
  nt.set_udf called for the (Filter, GroupBy) fused subtree.
                ↓
[Rust execute_filter_groupby fused entry]
  1. Wrap input Arrow buffers as MetalBuffer views (zero-copy).
  2. Dictionary-encode l_returnflag and l_linestatus → u32 codes.
  3. Run filter kernel on l_shipdate, producing keep-mask + scan idx.
  4. Composite-encode key columns (CPU-side, cheap).
  5. Run partitioned-hash build (A1) → row_to_group.
  6. Emit fused aggregation MSL kernel (cache by signature).
  7. Run aggregation pass → 8 output columns.
  8. Decode dictionary codes for output keys.
  9. Build result DataFrame.
                ↓
[Polars CPU runs Sort on the result]
                ↓
[Result returned to user]
```

## Error handling

Carries from M2:

- Unrecognized IR shapes → `Fallback`, ancestors poison, whole query routes to CPU.
- GPU kernel dispatch failures → `polars.exceptions.ComputeError`.
- No partial GPU work within a lifted subtree.

New for M3:

- **Cardinality estimator failure** at execution (TGSM overflow detected) → automatic A1→A2 re-dispatch within the same query. Logged at `debug=True` but not user-visible as an error.
- **Expression unfolding rejection** at plan time → `Fallback`, GroupBy subtree routes to CPU. Reason includes the unsupported expression shape.
- **String dictionary OOM** at execution (>2^32 unique strings, infeasible in practice) → `ComputeError`.
- **MSL template-engine compile failure** at first sight of a signature → `ComputeError` with the compilation error. Cached signatures are pre-validated.

Deliberate non-decisions:

- **No runtime correction** of bad routing decisions (other than the A1→A2 fallback).
- **No execution-time fallback to CPU** if GPU is routed and turns out slower.

## Testing strategy

Five layers from M2, extended.

### Layer 1: Kernel correctness — Rust proptest (`crates/polars-metal-kernels/tests/`)

Per new kernel, 256+ cases against pure-Rust reference. New test files:

- `test_groupby_partitioned.rs` — A1 build phase. Byte-equal `row_to_group` and `n_groups` against `cpu_partitioned_hash_reference`. Edge cases: TGSM-overflow trigger, hash collisions, all-same-key, all-unique.

- `test_chunked.rs` — multi-chunk correctness across rechunk and chunked-dispatch regimes; arbitrary chunk boundary patterns.
- `test_expression_unfolding.rs` — emitted MSL for arbitrary `sum(a*b)`-shaped expressions compiles, runs, matches CPU.

## Exit criteria

M3 done when all simultaneously true.

**Functional.**

1. `df.collect(engine=MetalEngine())` runs canonical TPC-H Q1 (Utf8 keys + inline `sum(a*b)` expressions) on GPU. Verified by `tests/python_integration/test_canonical_q1.py` and debug-log assertions.
2. High-cardinality groupby (10K–1M unique keys) routes through partitioned-hash (A1) below the cardinality threshold and sort-segment-reduce (A2) above. Verified by `MetalEngine(debug=True)` log assertions in `tests/python_integration/test_routing.py`.
3. A1 detects TGSM overflow at execution time and falls back to A2 without query failure. Tested by a synthetic worst-case fixture in `test_groupby_partitioned.rs`.
4. Fused multi-aggregation kernel: any query with N aggregations over M value columns dispatches exactly one Metal kernel for the agg pass. Verified by Metal capture count in benchmarks.
5. Filter routes GPU when its consumer is a GPU-routed GroupBy; otherwise CPU. Verified in `test_filter_groupby_fusion.py`.
6. String-key groupby works through the dictionary path (low cardinality) and the dedicated hash-kernel path (high cardinality). Both paths verified per cardinality regime in `test_string_groupby.py`.
7. Multi-chunk Series handled at the buffer bridge: rechunk-on-import below 256 MB, chunked-dispatch above. No `Fallback("multi-chunk")` for in-scope queries.
8. Smaller-integer composite keys (i8/i16/i32/u8/u16/u32) handled by the encoder. Composite-key width check still enforces ≤ 128 bits.
9. Binary expression unfolding in `.agg()` works for `*` `+` `-` `/` on column/column and column/literal pairs. Unsupported shapes fall back at plan time.

## Risks & open questions (M3-owned)

**Cardinality estimation is wrong.** The whole A1/A2 routing rests on HLL++ on a row sample. If the sample is unrepresentative, estimate skews → wrong build phase picked. *Mitigation:* A1 has the overflow-detect-and-bail mechanism (one extra dispatch on misroutes). *Open:* if misroutes are frequent, sample size scales up or estimator is replaced. M3's primary algorithmic risk.

**MSL template-engine compile latency.** Code-generating MSL per query signature means a `MTLLibrary` compile on first sight of each signature (~50–200 ms). Cache by signature hash amortizes across same-shape queries; first query of a session pays. *Mitigation:* pre-compile common signatures at module import. *Open:* if cold-start dominates a real workload, consider offline kernel-precompilation at wheel-build time.

**A2 sort kernel is technically a roadmap-item-6 implementation.** We chose to call it an internal helper; the algorithm is real radix sort. M4 inherits this kernel as either a starting point or competition. *Mitigation:* design A2's kernel signature so M4 can generalize it (parametric key dtype, not hardwired to u128). *Open:* M4 may decide to replace it.

**Dictionary encoding latency on large unique counts.** D Phase 1 is CPU-bound; for millions of unique strings it dominates. Phase 2 (dedicated hash kernel) addresses but adds a cardinality decision before encoding. *Mitigation:* sample-based unique-count estimate gates dict-vs-hash choice. *Open:* threshold tuning in microbenches.

**Filter+groupby fused subtree representation.** Two options: (a) introduce `FusedFilterGroupBy { filter_expr, groupby_spec }` as a new variant; (b) keep separate `Filter` and `GroupBy` nodes but mark them as co-routed via a `subtree_id` field. *Decision deferred to implementation plan.* Weak prior: (b), keeps IR shape stable across fused and unfused plans.

**100M Q1 memory pressure on smaller Macs.** The 100M-row Q1 fixture is ~9 GB in DRAM. Documented as M2-Ultra-only. Future Macs with smaller unified memory will OOM; the engine returns `ComputeError`, not silent slow-down. *Tracked in `docs/open-questions.md` § "Memory pressure on base M1"*; not blocking.

**MLX usage unchanged in M3.** No new MLX bindings; existing `cumsum_u8_to_u32` stays as-is. *Open:* if filter routing into GPU (C) wires filter compaction into the groupby pipeline, the cumsum step may benefit from the MLX-over-MTLBuffer rewrite the M1 retrospective flagged.

**Multi-chunk Series chunk-iteration kernels.** Every kernel that accepts a `MetalBuffer` will need a chunk-iterator variant, or the buffer bridge presents a synthetic single-buffer view. *Decision deferred to implementation plan.* Weak prior: synthetic view, minimizes kernel API churn.

**Polars rev drift.** Same risk as M1/M2; same mitigation (pin to `py-1.40.1`, bump at milestone boundary). *Open:* the Polars rev pinning policy may need revisiting if upstream changes the expression IR shape that G depends on.

**Q6's exact Polars IR shape is unverified.** `df.filter(...).select(expr.sum())` could land in py-1.40.1 as `Select(Filter, Agg)`, or as a distinct `Reduce` / `HSimpleAgg` node, or get folded into a `GroupBy` with empty keys by the optimizer. The walker dispatches all three shapes to the same `MetalPlanNode::GroupBy { keys: [], aggs: [...] }`, but recognizing the right IR node class is impl-time work. *Mitigation:* if Q6's IR shape turns out to be unrecognizable cheaply, the fallback is to land Q6 as a special-case `MetalPlanNode::Reduce` variant. Either path produces the same execution behavior.
