# M5 — Transparent acceleration of native opaque expressions (rolling + dt) — Design

**Date:** 2026-06-02
**Status:** approved design; ready for implementation planning (two sub-projects).
**Predecessor:** M4 (`docs/superpowers/specs/2026-05-28-m4-design.md`) — the MLX-subgraph-fusion engine, landed on `main` (PR #4).

## Goal

Make two *native* Polars expression families faster under `engine="metal"` — **`rolling_mean`/`sum`/`std`/`var`** and **`dt.year`/`month`/`day`** — with **no new user API**. The user keeps writing ordinary Polars; the engine accelerates it transparently and falls back to CPU for anything it doesn't match exactly.

This is the compute-intensity mission applied to two expression families M4 couldn't reach. It is explicitly *not* about inventing new vocabulary (that ruled out the deferred `dot`, `FFT`, and `corr` — see Non-goals).

## Background: the recognition gap M5 closes

M4's engine recognizes expressions at Polars' **post-optimization** `NodeTraverser`, whose `view_expression` exposes only a "core" set (Column, Literal, BinaryExpr, viewable Function, Cast, Agg, Ternary, Sort). **`rolling_*`, `dt.*`, list/array, `corr`, and `map_batches` all raise** — the walker can't see them (see `docs/open-questions.md` and the `m4-nodetraverser-opacity` memory). So today these fall back to CPU.

The escape hatch: `lf.serialize()` (the whole logical plan, **pre-optimization**) *does* expose them. M5 adds a recognition pass at that earlier point.

Two facts were verified during design (py-1.40.1):
- The cumsum-diff building blocks (`cum_sum`, `shift`, `fill_null`, `int_range(len) < lit`, `when(...).then(null).otherwise(...)`) are **all viewable** to the walker — so rolling stays inside the M4 fusion paradigm.
- Exact rolling semantics: first **w−1** rows are null (default `min_samples = window`); `var`/`std` are **sample (ddof=1)**; the naive `cum_sum − cum_sum.shift(w)` is **off-by-one** (it nulls the first full window).

## Scope

**In:** `rolling_mean/sum/std/var` (fixed integer window, F32, null-free, default options); `dt.year/month/day` (Date and tz-naive Datetime).

**Out (fall back to CPU, not errors):** tz-aware `dt`; `dt` fields beyond year/month/day; time-based / grouped / centered / weighted rolling; null-bearing inputs; non-F32 rolling columns.

**Deferred (not M5 — new-API, no native Polars equivalent):** list/array dot → matmul, correlation matrix, `Expr.fft()`, pairwise-distance kernels. A new API is acceptable for these *later*, but M5 is "accelerate existing," not "design new vocabulary."

## Architecture

A single **pre-optimization recognition pass**, invoked from the existing `LazyFrame.collect` monkey-patch *before* the engine callback (the only point where opaque expressions are visible via `lf.serialize`). The pass is a thin router over two **independent, pluggable recognizers**, each testable in isolation:

```
collect(engine="metal")
   │  (monkey-patch, pre-optimization)
   ▼
pre-opt recognition pass  ──reads lf.serialize()──▶ fast-reject if nothing matches
   ├─ rolling-rewrite  ── rewrite rolling_*(col,w) → cumsum-diff expr ─▶ existing M4 walker
   └─ dt-capture       ── recognize dt.*(col) → gregorian MSL kernel ─▶ fold-back column
   │  (no match / unsupported shape, dtype, option)
   ▼
unchanged LazyFrame ──▶ normal M4 engine / Polars CPU
```

**Hard fallback contract:** any unrecognized shape, dtype, or option leaves the LazyFrame untouched → Polars CPU. Correctness is never at stake; M5 only *adds* acceleration for exactly-matched shapes. The pass must fast-reject cheaply (the serialized plan is small — negligible per-collect overhead) and must not regress M4 paths.

**Two routing modes** reflect a real split between the features:
- **Rewrite-to-viewable** (rolling): there *is* a viewable arithmetic equivalent, so we rewrite and let the proven M4 engine run it. Low risk.
- **Capture + kernel + fold-back** (dt): there is *no* viewable equivalent, so we recognize, dispatch a new MSL kernel, and fold the result column back (reusing M4's HStack column stitch).

## Rolling design

**Ops/scope:** `rolling_mean`, `rolling_sum`, `rolling_std`, `rolling_var`; fixed integer window; F32; null-free; default options (`min_samples = window`, no centering, not time-based, not grouped). Else → CPU.

**Identities (cumsum-diff):**
- `S1 = cum_sum(x)`; `rolling_sum = (S1 − S1.shift(w).fill_null(0))`, then **null the first w−1 rows**.
- `rolling_mean = rolling_sum / w`.
- `S2 = cum_sum(x²)`; `rolling_var = (S2_diff − S1_diff² / w) / (w − 1)`; `rolling_std = sqrt(rolling_var)`; same head-mask.

(The `fill_null(0)` + explicit head-mask are required because `S1 − S1.shift(w)` nulls the first full window — the verified off-by-one.)

**Mechanism (two pieces, both within the M4 paradigm):**
1. **Pre-opt rewrite:** detect a handleable `rolling_*` in `lf.serialize`, rebuild that `with_columns`/`select` using the cumsum-diff expression.
2. **Walker vocabulary extension:** teach the analyzer + MLX subgraph the new ops the rewrite uses — `shift` (pad/slice), `fill_null`, row-index (`int_range`/`len`), and null-bearing `Ternary`. All verified viewable, so recognition is free; only MLX/host handling is new.

**Expected:** GPU cumsum (bandwidth-bound, ~`cum_sum`'s 4.5×) + cheap shift/arith → ~**10–15×** vs Polars CPU (125 ms at 10M, w=1000), and **window-size-independent** (vs Polars' O(n·w)).

**Risk:** matching Polars' edge + null semantics exactly is the whole game — heavy differential testing across window sizes and at the w−1 boundary. Null columns / non-default options → CPU.

## `dt` design

**Ops/scope:** `dt.year`, `dt.month`, `dt.day` on **Date** (i32 epoch-days) and **tz-naive Datetime** (i64 → days). tz-aware, other fields, null dates → CPU.

**The kernel (new MSL shader):** proleptic Gregorian `civil_from_days` (Hinnant-style), branchless integer math, one thread per row, no shared memory; i32 epoch-days → `(year, month, day)` i32 (Datetime divides to days first). One kernel file + a `tests/kernel/` correctness test (per repo conventions). Read the cuDF/candle reference for the date-math idiom before writing it.

**Mechanism (capture + dispatch + fold-back):**
1. **Pre-opt detect** `dt.year/month/day(col)` via `lf.serialize`, for the bounded shape (`scan → with_columns/select(dt.*) → simple tail`).
2. **Dispatch** the gregorian kernel on the materialized date column → an i32 column.
3. **Fold-back** that column into the frame — **reuses M4's HStack column stitch**. The novel parts are the kernel, wiring a **kernel-backed binding** (the first non-MLX dispatch in the fusion engine), and Date-vs-Datetime encoding handling.

**Expected:** Polars CPU `dt.year` ~178 ms at 10M (branchy); the kernel is memory-bound → ~**30–40×**. (If `dt.*` feeds a bandwidth-bound groupby the whole-query win is smaller; the bench measures the extraction in isolation, like M4's per-op benches.)

**Risk:** (a) kernel correctness across the full date range incl. leap years / negative epoch-days; (b) the new kernel-backed binding path; (c) Date-vs-Datetime encoding + the tz-naive-only restriction.

## Testing strategy (the three oracles)

- **Conformance:** Polars' own `rolling_*` and `dt` tests run under `engine="metal"` — they fall back to CPU today; after M5 they route and must stay byte-exact. The real bar.
- **Differential:** random windows / dtypes / dates incl. the w−1 boundary, empty, and **null** inputs (which must fall back); assert engine == CPU.
- **Kernel:** the gregorian shader, full date range (leap years, negative epoch-days).
- **CPU-parity gate:** the recognition pass adds ~0 overhead on non-matching plans.

## Perf gates

Flip the existing `_pending` `phase9_rolling_mean_w1000_10m` baseline entry; add a `dt_year_10m` entry. Both with `ratio_lt` gates (rolling ~10–15×, dt ~30–40×). M4's gates stay green.

## Non-goals

No new public API. No tz-aware dt, no dt fields beyond Y/M/D, no time-based/grouped/centered/weighted rolling, no null-input acceleration (CPU). corr/dot/FFT/pairwise remain deferred.

## Sequencing (this is the milestone design; each sub-project gets its own implementation plan)

1. **Rolling** first — lower risk, reuses the M4 engine (pre-opt rewrite + walker vocab extension). Ships a transparent win early. **This is the first implementation plan.**
2. **dt** second — the gregorian MSL kernel + kernel-backed binding + capture/fold-back. Its own plan once rolling lands.

## Open questions / risks

- **Plan-rewrite mechanics (rolling):** rebuild via the Python expr API from parsed pieces (simple shapes) vs JSON surgery + `LazyFrame.deserialize`. The plan picks one; parsed-pieces-for-bounded-shapes is the leaning.
- **Serialized-plan format stability:** semi-private, version-pinned to py-1.40.1 — a second pinned surface alongside the IR walker (acceptable, documented).
- **dt splice scope:** bounded to recognizable simple query shapes; broaden only if a real workload needs it.
- **`shift`/`fill_null` on MLX:** shift-with-null-fill is pad+slice (not circular roll) — confirm the MLX/host implementation matches Polars at the edges.

## Conventions (per CLAUDE.md)

No `unwrap()` outside tests; no `unsafe` outside `*-sys`/buffer crates (with `// SAFETY:`); errors surface as `polars.exceptions.ComputeError`; **null semantics match Polars exactly**; one MSL kernel per file with a matching `tests/kernel/` test; don't add a dependency without written justification.
