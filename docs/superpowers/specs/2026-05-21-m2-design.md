# M2 — Per-op routing layer + hash groupby — Design

**Status:** approved 2026-05-21, ready for implementation plan.

**Predecessor:** M1 (`docs/superpowers/specs/2026-05-20-m1-design.md`) landed scan / project / filter on GPU. The post-M1 perf investigation surfaced that the engine should be an *auto-router* on Apple Silicon's unified memory, not a GPU executor that handles everything. See `docs/open-questions.md` § "Routing layer: per-op GPU-vs-CPU dispatch on unified memory" for the architectural pivot motivation.

## Mission

Ship the per-op routing layer that turns `polars-metal` from "GPU executor for the supported subset" into "auto-router across CPU + GPU on Apple Silicon." Validate it end-to-end by landing hash groupby on GPU and demonstrating that a modified TPC-H Q1 (integer-encoded `l_returnflag` / `l_linestatus`, otherwise unchanged) beats pure-CPU Polars on M2 Ultra.

M2 conflates the routing-layer work with what was originally M3's hash-groupby scope. The single milestone makes sense because the routing layer needs a real workload that benefits from GPU to validate the architecture — and hash groupby is the first such workload.

## Scope (decisions locked)

### Workload validated

**Modified TPC-H Q1**, exactly:

```python
df.lazy()
  .filter(pl.col("l_shipdate") <= date_threshold)
  .group_by("l_returnflag", "l_linestatus")
  .agg(
      pl.col("l_quantity").sum().alias("sum_qty"),
      pl.col("l_extendedprice").sum().alias("sum_base_price"),
      pl.col("disc_price").sum().alias("sum_disc_price"),       # pre-projected
      pl.col("charge").sum().alias("sum_charge"),               # pre-projected
      pl.col("l_quantity").mean().alias("avg_qty"),
      pl.col("l_extendedprice").mean().alias("avg_price"),
      pl.col("l_discount").mean().alias("avg_disc"),
      pl.len().alias("count_order"),
  )
  .sort("l_returnflag", "l_linestatus")
  .collect(engine=MetalEngine())
```

**Modifications from strict TPC-H Q1:**
- `l_returnflag` and `l_linestatus` integer-encoded (i8 or i64 — both fit the fixed-width-key constraint). Polars `Utf8` columns are out of scope for M2.
- `disc_price` (= `l_extendedprice * (1 - l_discount)`) and `charge` (= `disc_price * (1 + l_tax)`) pre-projected into the input frame. The aggregation expression nesting (`sum(a * b)`) is deferred to M3 — see Hand-off § below.

**Test data:** 10M-row `lineitem`-shaped table, generated by a deterministic Python fixture (no external dbgen dependency). The fixture preserves TPC-H Q1's analytical shape: `l_returnflag` ∈ {0, 1} (encoded `A`/`N`), `l_linestatus` ∈ {0, 1} (encoded `F`/`O`), realistic ranges for the numeric columns, deterministic seed so benchmarks are reproducible across runs.

### IR nodes handled natively on GPU

The walker now recognizes four IR node shapes (three from M1, one new):

- **DataFrameScan** — unchanged from M1.
- **SimpleProjection / Select** (metadata-only) — unchanged from M1.
- **Filter** — unchanged from M1, but **routing layer now sends it to CPU by default** at all input sizes. Kernels stay shipped; the router decides not to call them. See § Cost model below.
- **GroupBy** — NEW. Hash-based two-pass groupby on fixed-width composite keys, with up to 5 aggregations (sum / mean / count / min / max).

All other IR node types fall back, same as M1.

### Routing decisions (cost model)

The cost model is a constants table in Rust, refined by PR as kernels evolve. Starting thresholds:

| Op | Decision | Threshold |
|---|---|---|
| Filter | CPU | always (filter is memory-bound; GPU has no fundamental advantage on unified memory) |
| Project / SimpleProjection | follow input | inherit child's decision (metadata-only on our side either way) |
| Scan | follow output | inherit parent's decision (input-side bridge is zero-copy either way) |
| GroupBy | GPU iff `n_rows > 100_000` | tuned from microbenches |

**Affinity smoothing.** After the bottom-up decision pass, a second pass smooths transitions where costs are close (within 20% of the alternative). The intent: minimize unnecessary GPU↔CPU transitions even though the handoff is zero-copy on unified memory — keeps debug logs simpler and avoids future-overhead surprises if M3 adds non-trivial handoff costs.

### Type matrix

- **Group keys:** fixed-width only — i8 / i16 / i32 / i64 / f32 / f64 / bool. Multiple keys composite-encoded into a single u128 (see § Composite key encoding).
- **Aggregation value columns:** i64 / f64 (M1's matrix). M1's bool support carries over (count works on any column; bool can be cast to i64 for sum-as-count-of-true).
- **Out of scope for M2:** variable-width keys (strings), >128-bit composite widths, smaller-integer aggregations (i8/i16/i32). All fall back cleanly.

### Aggregations delivered

Six entry points: the 5 value-column aggregations + `pl.len()`.

| Op | Reads value column? | Null behavior |
|---|---|---|
| `sum` | yes | nulls excluded; sum of empty group → 0 (matches Polars CPU) |
| `mean` | yes | nulls excluded; mean of empty group → null |
| `count` | yes (to skip nulls) | counts non-null values |
| `min` | yes | nulls excluded; min of empty group → null |
| `max` | yes | nulls excluded; max of empty group → null |
| `pl.len()` | no | row count including nulls; one kernel that reads only `row_to_group` |

`pl.len()` is implemented as a separate row-count kernel (it doesn't touch a value column). Q1 uses it for `count_order`.

### MLX usage

No new MLX ops in M2. The two-pass groupby is custom MSL throughout.

The existing MLX `cumsum_u8_to_u32` binding (from M1) is unchanged.

## Architecture

### Three-layer flow

```
Polars LazyFrame.collect(engine=MetalEngine())
         │
         ▼
Walker (Python) ── translates Polars IR to MetalPlanNode tree
         │            for every recognized op. No routing decisions.
         │            Hands the tree to Rust via
         │            _native.compute_lifting_plan.
         │
         ▼
Router (Rust)  ── walks the MetalPlanNode tree, consults the cost
         │            model, runs the affinity smoothing pass,
         │            returns a LiftingPlan: per-subtree GpuLift /
         │            CpuLeave / Fallback decisions.
         │
         ▼
Walker (Python) ── applies the lifting plan. For each GpuLift
         │            subtree calls nt.set_udf(udf_for_<op>).
         │            For CpuLeave subtrees does nothing — Polars'
         │            CPU executor runs the subtree as-is.
         │
         ▼
Dispatch       ── GPU subtrees execute via Rust (existing M1 filter
                  pipeline + new groupby pipeline). Buffers cross
                  the GPU↔CPU boundary as zero-copy MTLBuffer/Arrow
                  views (unified memory).
```

The walker stays minimal: pattern-matches IR nodes, builds a uniform `MetalPlanNode` tree. The router has all the policy; the walker has all the IR plumbing.

### MetalPlanNode extension

The plan IR enum gains one variant in `crates/polars-metal-core/src/plan/mod.rs`:

```rust
MetalPlanNode::GroupBy {
    input: Box<MetalPlanNode>,
    keys: Vec<(String, MetalDtype)>,      // column name + dtype
    aggs: Vec<AggSpec>,                    // (input_col, agg_op, output_alias)
    // composite-key encoding info computed at dispatch time, not stored here
}
```

The walker fills `keys` and `aggs` from the Polars `GroupBy` IR node. Rust computes composite key encoding on demand.

### LiftingPlan data structure

Returned by `_native.compute_lifting_plan(plan_dict)`. Per-node decision:

```rust
enum NodeDecision {
    GpuLift,                  // walker should call nt.set_udf
    CpuLeave,                 // walker leaves this subtree to Polars
    Fallback(String),         // unrecognized; whole subtree falls back
}

struct LiftingPlan {
    decisions: HashMap<NodeId, NodeDecision>,
    // serialized to Python as a dict on the wire
}
```

The walker traverses its own IR tree, looks up each node's decision by id, and applies it.

### Composite key encoding

CPU-side preprocessor in `polars-metal-kernels/src/groupby.rs`:

```rust
fn encode_keys(key_cols: &[KeyColumn]) -> (Vec<u128>, KeySchema)
fn decode_keys(encoded: &[u128], schema: &KeySchema) -> Vec<DecodedColumn>
```

Encoder packs each key column's bytes into successive bit ranges of a u128, recording widths and offsets in a `KeySchema`. Decoder reverses on the result side.

Width check at plan time: if the sum of key column widths > 128 bits, `Fallback("key columns total {N} bits; M2 supports ≤ 128")` and the GroupBy subtree routes to CPU.

### Two-pass groupby algorithm

Ported from `references/cudf/cpp/src/groupby/hash/groupby.cu`. The algorithm:

```
Inputs:
  encoded_keys: MetalBuffer<u128>[n_rows]
  value_cols:   N value columns + validity bitmaps
  aggs:         which aggregations per value column

Pass 0 (hash):
  hashes: MetalBuffer<u32>[n_rows]
  One MSL kernel, one thread per row. Hashes each composite key
  via a mixing function (xxhash-style; deterministic so output
  row order is reproducible across runs at the same input).

Pass 1 (build):
  Allocate hash_table_size = next_pow2(n_rows / load_factor).
  Slots hold (key_hash, group_id, count_so_far).
  Each row: atomic-CAS to find-or-insert into the table; receives
  a group_id. Output:
    row_to_group:        MetalBuffer<u32>[n_rows]
    group_count:         scalar u32
    first_row_per_group: MetalBuffer<u32>[group_count]
                         (for reconstructing key values in the
                         output, since the hash table only stores
                         hash + group_id, not the original keys)

Pass 2 (aggregate):
  For each (value_col, agg_op):
    One MSL kernel. Thread per row: atomic-OP onto
    output[group_id]. Null-aware: skip null rows for sum / mean /
    min / max; count_non_null increments only for non-null rows;
    pl.len() counts every row.
  Outputs: one MetalBuffer per aggregation, length = group_count.

Pass 3 (result reconstruction, CPU-side):
  - Read encoded keys at first_row_per_group indices.
  - Decode composite keys → per-key-column Vec<T>.
  - Build result DataFrame: group_count rows ×
    (k key cols + agg_count agg cols).
```

The hot path is the build phase (atomic-CAS contention on duplicate keys). For Q1's degenerate case (4 unique groups, 10M rows), contention is extreme; the design relies on Metal's atomic primitives being fast and on retry-on-collision being cheap relative to non-atomic alternatives. If profiling shows the build phase bottlenecking, M2 may revisit with a sort-then-scan algorithm (cuDF has both implementations).

### Data flow for modified Q1

```
[Polars IR] ─────────────────────────────────────────────────
   DataFrameScan
   ↓
   Filter (l_shipdate <= threshold)
   ↓
   GroupBy (keys=[returnflag, linestatus], aggs=[8 specs])
   ↓
   Sort (key cols)
[Walker translates IR → MetalPlanNode tree (all 4 nodes)]
                ↓
[Rust router computes LiftingPlan]
  Scan:    CpuLeave  (inherits Filter's decision)
  Filter:  CpuLeave  (rule: filter always CPU)
  GroupBy: GpuLift   (10M rows > 100K threshold)
  Sort:    CpuLeave  (sort not yet implemented in router)
[Walker applies lifting plan]
  nt.set_udf is called only for the GroupBy node.
  Other nodes left for Polars' CPU executor.
                ↓
[Polars CPU runs Scan + Filter natively]
  Output: filtered Polars DataFrame, Arrow buffers in shared DRAM.
                ↓
[GroupBy UDF in _udf.py]
  Calls _native.execute_groupby(df_pydf, plan_dict).
                ↓
[Rust execute_groupby]
  1. Wrap input Arrow buffers as MetalBuffer views (zero-copy).
  2. Composite-encode key columns (CPU-side, cheap).
  3. Run two-pass groupby pipeline (hash + build + aggregate).
  4. Decode composite keys; build result columns.
  5. Return result DataFrame.
                ↓
[Polars CPU runs Sort on the result]
                ↓
[Result returned to user]
```

The first user-visible win: M2 demonstrates that the engine routes per op, and the result for `Q1` arrives faster than `engine="cpu"` despite filter and sort both running on CPU.

## Error handling

Carries over from M1:
- Unrecognized IR shapes → `Fallback`, ancestors poison, whole query routes to CPU.
- GPU kernel dispatch failures → `polars.exceptions.ComputeError`.
- No partial GPU work within a lifted subtree.

New for M2:
- **Composite key overflow** at plan time → `Fallback`, GroupBy subtree routes to CPU.
- **Hash table OOM** at dispatch time → `ComputeError`. Size is `next_pow2(n_rows / load_factor)`, so no runtime overflow can happen by construction; only true OOM on memory-constrained Macs surfaces this.
- **Lifting plan inconsistency** (walker can't match plan IDs to IR nodes) → defensive `ComputeError` with the inconsistency described. Catches Rust/Python protocol drift loudly.

Deliberate non-decisions:
- **No runtime correction** of bad routing decisions. Cost model is offline-tuned.
- **No execution-time fallback.** If GPU is routed and turns out slower, we follow through. Tuning happens by PR.
- **No partial subtree lifting.** Same all-or-nothing rule as M1.

## Testing strategy

Five layers, with a unified taxonomy.

### Layer 1: Kernel correctness — Rust proptest (`crates/polars-metal-kernels/tests/`)

**Property-based testing happens in Rust against pure-Rust reference implementations.**

`test_groupby.rs` — proptest at 256 cases per property:
- Per (aggregation × null-density bucket × key-cardinality bucket): GPU output equals `cpu_groupby_reference(...)` byte-exact.
- Edge cases (empty, single-row, all-same-key, all-unique, constructed hash collisions) included as explicit cases.

`test_filter_proptest.rs` — migration of M1's `tests/diff/test_filter_random.py`. Property: random (predicate, source-column) pairs run through the compaction pipeline match a pure-Rust reference `cpu_filter_compact_reference`. Same shape as the existing kernel proptests.

### Layer 1.5: Router behavior — Rust unit tests (`crates/polars-metal-core/tests/test_router.rs`)

Synthetic `MetalPlanNode` trees → assert `compute_lifting_plan` returns the expected `LiftingPlan`. Covers:
- Each cost-model rule (filter→CPU, groupby→GPU at threshold, etc.)
- Affinity smoothing on close-cost transitions
- Fallback propagation
- Composite-key width check at plan time

### Layer 2: Explicit Python tests — `tests/python_integration/`

One assertion per named case. Not property-based; covers engine-boundary correctness.

`test_routing.py` — verifies routing decisions per IR node via `MetalEngine(debug=True)` debug logs. For each test query, parse the logs and assert the expected (CpuLeave / GpuLift) per node.

`test_groupby.py` — explicit cases:
- Empty input, single-row → single-group, all-same-key, all-unique-keys
- Multi-key (Q1 shape: 2 i64 keys, also tested with 3 mixed-type keys)
- Null in key column → null becomes its own group (matches Polars)
- Null in value column → aggregations skip nulls
- Engine result matches CPU result via `assert_frame_equal` (byte-exact)

`test_filter_edges.py` — migrated from `tests/diff/test_filter_edges.py`. Same cases; new directory.

### Layer 3: Conformance — upstream Polars test suite (`tests/conformance/`)

Wire two new upstream paths into the harness:
- `tests/unit/operations/test_group_by.py`
- `tests/unit/operations/aggregation/` (the matching files in `py-1.40.1`)

Same model as M1's T22: per-path baseline of pre-existing CPU failures; assert no NEW failures under `engine=MetalEngine()`. Out-of-scope tests (string-key groupby, window functions, etc.) live in the baseline; in-scope tests must pass.

### Layer 4: Benchmarks — `tests/bench/` and `benches/`

**Criterion (per-kernel):**
- `benches/groupby_hash.rs`, `benches/groupby_build.rs`, `benches/groupby_aggregate.rs`. Sizes 100K / 1M / 10M / 100M. Null densities 0% / 50% / 100% on value columns.

**pytest-benchmark (end-to-end):**
- `tests/bench/test_tpch_q1.py` runs the modified Q1 query under both engines.
- `baseline.json` gets a new `tpch_q1_modified` entry with `cpu_ms`, `metal_ms`, `ratio_metal_over_cpu`.

**M2 perf gate:** `tpch_q1_modified` `ratio_metal_over_cpu < 1.0` on M2 Ultra. Filter ratio is exactly 1.0 by definition (routed to CPU); documented in `baseline.json`'s `_notes`.

### Layer 5: Portability — manual at milestone boundary

`make gate` on M2 Ultra (dev), M2 16 GB, M1 8 GB. The 10M-row Q1 has a 16-column lineitem table; memory pressure on the 8 GB box may surface tuning needs. Capture results in the M2 retrospective.

## Exit criteria

M2 is done when all of the following are simultaneously true.

**Functional.**

1. `df.collect(engine=MetalEngine())` on any query containing a hash-groupby with fixed-width composite keys (≤ 128 bits total) runs the groupby on GPU. Verified by `MetalEngine(debug=True)` log assertions in `tests/python_integration/test_routing.py`.
2. The router produces a per-IR-node lifting plan and the walker applies it correctly. Filter routes to CPU; groupby routes to GPU at row counts above the threshold; affinity smoothing kicks in for close-cost transitions.
3. Composite-key groupby for any number of keys totaling ≤ 128 bits handled natively; wider key sets cleanly fall back via `Fallback` at plan time.
4. All 6 aggregation entry points supported with null-aware Polars semantics: sum, mean, count, min, max, plus `pl.len()` (row count).
5. Any unsupported shape (string keys, wider composite, joins, unsupported aggregation expressions) falls back cleanly. No partial GPU work.
6. `polars.exceptions.ComputeError` for any category-B failure.

**Correctness.**

7. `make test-kernel` passes: groupby proptest green at 256 cases per (aggregation × null-density × key-cardinality) bucket; filter proptest migrated from M1's hypothesis test passes.
8. `make test-unit` passes including new router unit tests.
9. `tests/python_integration/` tests pass (`test_groupby.py`, `test_routing.py`, migrated `test_filter_edges.py`).
10. `make test-conformance` passes with zero new failures vs CPU baseline, including the newly-wired upstream groupby paths.
11. `tests/diff/` directory removed.

**Performance.**

12. **Headline.** `tests/bench/baseline.json` records `tpch_q1_modified` with `ratio_metal_over_cpu < 1.0` on M2 Ultra. Full Q1 (CPU-routed filter + GPU-routed groupby + CPU-routed sort) strictly faster than pure-CPU Polars.
13. Per-kernel criterion benches added for groupby. No regression on M1's existing kernels (filter, cmp, logical, scatter).
14. Router plan-time overhead under 1 ms for any query of practical size.
15. Filter ratio is 1.0 by definition (routed to CPU); documented in `baseline.json` notes.

**Quality.**

16. `make gate` passes locally on M2 Ultra.
17. Portability gate passes on M2 (16 GB) and M1 (8 GB); results in M2 retrospective.
18. Lint clean (clippy, fmt, ruff check, ruff format).

**Documentation.**

19. `docs/architecture.md` extended: routing layer, lifting-plan exchange, groupby pipeline.
20. `docs/kernel-authoring.md` extended: two-pass count-then-fill groupby idiom, per-aggregation kernel pattern.
21. `docs/open-questions.md` updated: M2-resolved items struck through; new M2-surfaced items added.
22. M2 retrospective written in this spec's closing section.
23. Testing taxonomy documented (where property tests live, where explicit cases live).

**Hand-off to M3.**

24. Routing pivot in production. M3 inherits the per-op routing model.
25. Variable-width keys (strings) deferred to M3 or later.
26. Multi-aggregation expression unfolding (`sum(a * b)` shape) deferred to M3.
27. Join operators deferred to M3+.

## Risks & open questions (M2-owned)

**Hash-table contention on hot keys.** Q1's degenerate case (4 unique groups, 10M rows) means every row contends on 1 of 4 hash-table slots. Atomic-CAS retry-loop performance under extreme contention is the dominant unknown. *Mitigation:* if microbenches show >2× degradation vs ideal, M2 revisits with sort-then-scan algorithm (cuDF has both). Tracked as M2's primary algorithmic risk.

**Cost-model threshold tuning.** Starting thresholds are educated guesses; real M2 microbenches will move them. Cost data and the implementation live in the same Rust module so they evolve together by PR.

**Affinity smoothing semantic.** First implementation uses "within 20% of alternative." If this leads to bad decisions, the threshold is one PR; the algorithm shape (single second-pass smoothing) doesn't need to change.

**Multi-chunk Polars Series.** M1 assumed single-chunk per Series; M2 inherits this. If a real-world DataFrame arrives chunked, the walker doesn't probe — explicit failure or silent wrong-result is possible. *Mitigation:* defensive check at scan time, error if chunk count > 1. Full multi-chunk support is M3+.

**Polars rev drift.** Same risk as M1; same mitigation (pin to py-1.40.1, bump at milestone boundary).

**Surprises during execution.** Plan-vs-reality deltas.

- **Apple Silicon Metal lacks 64-bit atomic ops (T16/T19/T22 all affected).** The plan assumed CUDA-like atomic primitives. Reality on toolchain 32023.883: `atomic_long` is "unknown type name"; `atomic_ulong` exists but `atomic_fetch_add_explicit` and `atomic_compare_exchange_weak_explicit` template-fail for it. Only `atomic_uint`/`atomic_int` (32-bit) have the full atomic op set. Also: `double` is unavailable in compute kernels (M1 already knew this for cmp_f64). Forced a redesign of aggregate kernels into a hybrid 32-bit GPU + 64-bit CPU-finalize scheme. Documented in detail in `docs/kernel-authoring.md` § "Apple Silicon Metal atomic ops constraint". Discovered during T16 atomic-helpers prototyping, confirmed by direct compile probe in T19.
- **SIMD-group lockstep execution rules out spin-wait designs (T19).** The build kernel's first design used "claim slot → spin until READY bit set". This deadlocks because threads in the same SIMD warp execute in lockstep — the waiting thread prevents the writer from running. Required two redesigns: first a "skip CLAIMED slots; track saw_claimed" pattern (commit `2a6b02a`), then ultimately a pivot of the build phase to CPU HashMap (commit `2d60b23`). The proptest surfaced the bug; the unit tests passed for low-cardinality cases because contention was low enough to mask it.
- **GroupBy build phase shipped on CPU, not GPU (T19 retrospective).** Spec § "Two-pass groupby algorithm" assumed GPU atomic-CAS find-or-insert. Reality: the concurrency primitives needed (64-bit atomics + non-lockstep spin) aren't available together on this hardware. The shipped design uses `std::collections::HashMap` in Rust for the build phase. For Q1's degenerate shape (4 groups × 10M rows), CPU build touches only 4 hash table entries and is trivially fast. For high-cardinality groupby, this is the largest M3 open question.
- **Q1's 2 × i64 key encoding doesn't fit the 128-bit budget (T36).** Spec § "Workload validated" said "i8 or i64 — both fit the fixed-width-key constraint". The encoder reality: 1 (null bit) + 64 (i64 data) = 65 bits/key; 2 × 65 = 130 > 128. Q1's fixture uses **Boolean** for `l_returnflag`/`l_linestatus` (2 × 2 bits = 4 bits, well under budget); documented in `tests/bench/_lineitem_fixture.py`'s docstring. M3 can address by adding i8/i16/i32 to `KeyDtype`.
- **Pre-existing `ShaderLibrary::load` bug exposed by serial test execution (Phase 6 follow-up).** The aggregate proptests need single-threaded execution because of Metal command-queue contention; switching to `--test-threads=1` then surfaced a bug where `ShaderLibrary::load` deleted its temp metallib file after first use, breaking subsequent loads (the OnceLock cached the path; the file was gone). Production was unaffected (cached `ShaderLibrary` reused). Fixed in commit `a212012` by removing the eager file delete; OS cleans up on process exit.
- **Plan API references repeatedly off (T16/T17/T18/T19/T22/T27/T38).** The plan's pseudocode referenced `MetalDevice::new()`, `device.new_buffer_with_bytes`, `device.pipeline_for`, `queue.compute_encoder`. The actual API is `MetalDevice::system_default()` (from `polars_metal_buffer`), `device.new_buffer_from_bytes`, `shared_library(&device).pipeline(name)`, `queue.dispatch_1d(&pso, &[&buf, ...], n)`. Subagent prompts had to redirect each time. Lesson: future plans should read actual existing API rather than inferring from M1 doc conventions.
- **Polars py-1.40.1 GroupBy IR shape differed from plan's pseudocode (T11).** The plan assumed `Alias` expression nodes wrap `Agg` expressions; reality is `agg_expr.output_name` on the `PyExprIR` wrapper itself, no `Alias` node. `Agg.name` is lowercase ("sum"), not CamelCase. `Agg.arguments` contains raw `int` arena indices, not `PyExprIR` objects. Discovered via runtime diagnostic in the T11 implementer; documented in `_walker.py` comments.
- **`_callback.py::_strip_side_channels` field-extraction differed in production (T7 follow-up).** The plan used `len(plan.get("df", []))` for row counts; the user/linter updated to `df.height()` with try/except defensive guarding. Better in practice (PyDataFrame's `__len__` semantics aren't documented stable). No retrospective concern.

**Still to revisit at M3** (not blockers — design inputs for the next milestone).

- **Apple Silicon 64-bit atomics catch-up.** When the Metal toolchain ships `atomic_long`/`atomic_ulong` with full op set, add native-64 GPU aggregate kernels behind a build-time probe + runtime feature flag. `docs/open-questions.md` § "Apple Silicon Metal 64-bit atomics gap".
- **GPU build phase for high-cardinality groupby.** CPU HashMap ships in M2 and is fine for Q1 / low-cardinality. For 1M+ unique groups, a sort-then-segment-reduce design (cuDF has one) becomes attractive. `docs/open-questions.md` § "GroupBy build phase on CPU".
- **String-key groupby.** Out of scope for M2. M3 needs to decide between dictionary-encoding at the buffer bridge vs a dedicated string-hash kernel (likely both). `docs/open-questions.md` § "String-key groupby".
- **Multi-chunk Series support.** M2 falls back on chunked Series; M3 should land proper multi-chunk handling at the buffer-bridge layer. `docs/open-questions.md` § "Multi-chunk Series support".
- **Composite key i8/i16/i32 dtypes.** M2 encoder supports i64/f64/bool. Adding smaller integer dtypes would let Q1 use its naturally-typed `l_returnflag`/`l_linestatus`-as-i8 instead of Boolean. `docs/open-questions.md` § "Composite key 128-bit limit".
- **Conformance baseline drift automation.** Manual recapture today; consider automation when `make refresh-refs` bumps the pinned Polars rev. `docs/open-questions.md` § "Conformance baseline drift".
- **Multi-aggregation expression unfolding** (`sum(a * b)`). Spec deferred this to M3; pre-projected columns ship in M2's Q1 fixture. M3 needs an expression rewriter.
