# M4 — Compute-intensity pivot: MLX subgraph fusion, cumsum-diff rolling, list-dot vector search, `Expr.fft()` — Design

**Status:** draft 2026-05-28, awaiting sign-off.

**Predecessor:** M3 (`docs/superpowers/specs/2026-05-22-m3-design.md`) landed canonical TPC-H Q1, Q6, and the walker that recognizes their shapes. The honest post-M3 measurement (see `tests/bench/baseline.json`, written 2026-05-27) showed Metal **2.83–19.6× slower** than Polars CPU on the TPC-H slice — the pivot trigger. The M4 benchmark survey (`docs/m4-benchmark-survey.md`, 2026-05-28) measured 15-of-15 wins of 4–80× on the compute-shaped workload class and identified the new direction. CLAUDE.md was revised to make the pivot canonical (Mission, Non-goals, Architectural principles, Roadmap, headline benchmark suite).

This spec covers **Phases 8–11** of the revised roadmap — the chunk of M4 work that shares one infrastructure piece (MLX subgraph fusion) and produces four measurable headline wins. Phases 12 and 13 (custom MSL gregorian-calendar kernel; pairwise string / time-series distance) are deferred to a follow-on chunk because they are different work (new MSL kernels, not MLX routing).

## Mission

Build the polars-metal walker into an **MLX-subgraph-fusion engine for compute-shaped expression trees**, and deliver four headline wins on M2 Ultra that the engine now serves end-to-end via `df.collect(engine="metal")`:

| Phase | Workload                                       | Measured MLX ceiling | M4 target wall-clock | Polars CPU baseline |
|-------|-----------------------------------------------|----------------------|----------------------|---------------------|
| 8     | Black-Scholes-shape F32 transcendental chain  | 3.86 ms              | < 6 ms               | 242 ms              |
| 8     | Haversine 10M rows                            | 3.49 ms              | < 6 ms               | 181 ms              |
| 8     | F32 std / var global reductions               | 1.12 ms              | < 2 ms               | 7 ms                |
| 8     | F32 sort 10M rows                             | 7.88 ms              | < 10 ms              | 33 ms               |
| 8     | F32 top-k 10M K=100                           | 7.57 ms              | < 10 ms              | 97 ms               |
| 8     | F32 cumsum 10M rows                           | 5.67 ms              | < 8 ms               | 37 ms               |
| 8     | Correlation matrix 200 × 200k                 | 16.7 ms              | < 22 ms              | 131 ms              |
| 9     | rolling_mean W=1000 over 10M F32 (cumsum-diff)| 5.83 ms              | < 8 ms               | 114 ms              |
| 9     | rolling_sum  W=1000 over 10M F32              | (= rolling_mean)     | < 8 ms               | ~114 ms             |
| 10    | Cosine top-k Q=100 N=100k D=768 (`Array[F32, 768]`)| 4.94 ms          | < 8 ms               | ~56,000 ms (proj)¹  |
| 10    | L2 k-NN Q=100 N=1M D=128 (SIFT1M shape)       | 36.8 ms              | < 50 ms              | (no Polars baseline)|
| 11    | FFT 8M-point 1D F32 via `Expr.fft()`           | 1.60 ms              | < 3 ms               | (no Polars API)     |

¹ Polars-native columnar formulation; users today escape to NumPy (145 ms at this scale).

The wall-clock targets allow for FFI marshalling, expression-graph build, MLX `eval()`, and result fold-back overhead beyond the raw MLX time. They are tight, deliberately: the architectural principles in CLAUDE.md require CPU parity on the non-compute glue, and the targets enforce it. If we ship a 50× compute win that collapses to 5× because of overhead, we've broken the design contract.

Reframed from M3: M3 proved "the engine can route a TPC-H query end-to-end" (and learned that the workload class was wrong). M4 proves "the engine can route a compute-shaped expression tree end-to-end and the compute win shows up in wall-clock."

**Perf gate per phase:**

- Phase 8 — every workload in the table above lands within its M4 target. The fused-subtree path delivers the win; the surrounding ops (filter, project, materialize) add ≤ 5% overhead vs Polars CPU on those ops alone.
- Phase 9 — rolling_mean and rolling_sum hit their targets at all window sizes (100, 1000, 10000).
- Phase 10 — cosine top-k at both N=100k and N=1M scales; L2 k-NN at SIFT1M scale.
- Phase 11 — `pl.col("x").fft().collect(engine="metal")` works as a public API and hits the FFT target.

### IR / expression shapes handled natively on GPU

The walker now recognizes **compute subtrees** within `Select`, `HStack`, and `WithColumns` IR nodes. A compute subtree is a maximal contiguous chain of supported expressions whose root is the column-being-built expression.

Supported expression op set, M4 chunk (Phases 8–11):

**Element-wise F32 (Phase 8):**
- Arithmetic: `+`, `-`, `*`, `/`, `%`, `abs`, `neg`
- Comparison (returning Bool): `==`, `!=`, `<`, `<=`, `>`, `>=`
- Logical: `&`, `|`, `~` (only on Bool intermediates from comparisons)
- Conditional: `when(...).then(...).otherwise(...)` with arbitrarily-nested cascades
- Transcendental: `sin`, `cos`, `tan`, `sinh`, `cosh`, `tanh`, `arcsin`, `arccos`, `arctan`, `arctan2`
- Log/exp: `log`, `log2`, `log10`, `log1p`, `exp`, `exp2`
- Power: `sqrt`, `cbrt`, `pow`, `square`
- Rounding: `floor`, `ceil`, `round`
- Cast between F32 ↔ F64 (and I32 → F32 for narrow inputs)
- Literal F32 and Bool constants

**Reductions (Phase 8 terminus):**
- `sum`, `mean`, `min`, `max`, `std`, `var`
- `n_unique` — NOT supported in this chunk (deferred)
- `argmin`, `argmax`
- Single-global `quantile(p)` (not rolling)

**Ordered ops (Phase 8 terminus):**
- `sort(asc=...)`, `top_k(k)`, `bottom_k(k)`
- `cum_sum`, `cum_max`, `cum_min`, `cum_prod`

**Matmul-shaped (Phase 8, Phase 10):**
- `(N, K) @ (K, M)` via `Array[F32, K].dot(lit)` or `List[F32].eval(element() * lit).list.sum()`

**Rolling (Phase 9):**
- `rolling_mean(window_size=W)`, `rolling_sum(window_size=W)` — expressed internally as cumsum-diff. Window size must be a compile-time literal at plan time (not a runtime expression).
- `rolling_var(window_size=W)` — via cumsum and cumsum² (also expressible as cumsum-diff). `rolling_std` follows trivially.

**FFT (Phase 11):**
- New API surface: `pl.Expr.fft()` registered via Polars' expression-extension hook. Operates on F32 (or castable-to-F32) numeric column. Output is `Complex[F32]` — but Polars has no complex dtype; result is a 2-column struct `(real F32, imag F32)`. See "Algorithm details — FFT API" below.

**NOT in scope this chunk (deferred to Phases 12+):**
- String ops, regex.
- Datetime `dt.year/month/day` decomposition (gregorian calendar custom MSL kernel — Phase 12).
- Pairwise string distance (Levenshtein, DTW — Phase 13).
- Hash groupby, hash join (M2/M3 conformance only).
- Sort-merge join.
- Window functions with `partition_by` (use Phase 9 cumsum-diff for the partition-free shapes that matter).

### Routing decisions (compute-density cost model)

The walker adds a **compute-density estimator** to its routing logic. For each candidate subtree:

```
density = SUM over ops in subtree {
              op.flops_per_row × n_rows
          } / SUM over input columns {
              col.bytes_read
          }
total_flops = SUM over ops in subtree {
                  op.flops_per_row × n_rows
              }
```

Subtree routes to GPU iff:

```
total_flops > MIN_FLOPS_THRESHOLD          (default: 5e7)
AND
n_rows > MIN_ROWS_THRESHOLD                (default: 1e5)
```

The FLOP estimates are static per-op constants (per op, F32 unit cost): `+ - *` = 1; `/` = 4; `sin/cos/exp/log` = 10; `sqrt` = 4; `tanh` = 12; matmul = `2 * K`; reduction = 1; sort = `log2(n)`; cumsum = 2. These are not precise; they only need to be approximately ordered so the threshold decides correctly. Calibration is a one-time pass at M4 close; we don't tune per workload.

Below threshold, fall back to CPU. The threshold ensures we don't pay GPU dispatch overhead (5–15 ms) on workloads that finish in 1 ms on CPU.

| Op category                  | Compute-density routing decision | Reasoning                                  |
|------------------------------|----------------------------------|--------------------------------------------|
| Single arithmetic on F32     | CPU (below threshold at any N)   | CPU NEON wins; dispatch overhead dominates  |
| 5+ ops F32 chain at 10M+ rows| GPU                              | Crosses density threshold                  |
| Black-Scholes-shape 6+ ops   | GPU                              | High density, easy win                     |
| F32 sort at 1M+ rows         | GPU                              | log2(N) work; MLX radix wins               |
| F32 sort at 10k rows         | CPU                              | Below row threshold                        |
| List/Array dot(lit)          | GPU                              | Matmul shape, density very high            |
| Rolling mean N≥100k          | GPU                              | cumsum-diff wins decisively                |
| Hash groupby                 | CPU (M2/M3 conformance only)     | Per Mission / Non-goals                    |
| String ops                   | CPU                              | Per Non-goals                              |

Routing decisions are computed in `crates/polars-metal-core/src/router/compute_density.rs` (new). The output is a `FusionScope` — a marked subtree with the recognized op list, estimated total FLOPs, identified input/output columns. The Rust router consumes this and emits a `MetalPlanNode::FusedExprGraph { scope, ... }` plan node.

### MLX FFI surface (extended)

`crates/polars-metal-mlx-sys` today exposes a single binding (`cumsum_u8_to_u32`). M4 extends this with the full MLX surface needed for the Phase 8 op set. Bindings group naturally:

```
mlx_sys::elementwise   sin cos tan sinh cosh tanh asin acos atan atan2
                       log log2 log10 log1p exp exp2
                       sqrt cbrt pow square abs neg
                       floor ceil round
                       add sub mul div mod
                       eq ne lt le gt ge
                       logical_and logical_or logical_not
                       where  (= when/then/otherwise)
                       cast   (dtype → dtype)
mlx_sys::reduce        sum mean min max std var argmin argmax
mlx_sys::sort          sort argpartition (for top-k)
mlx_sys::scan          cumsum cumprod cummax cummin
mlx_sys::matmul        matmul (the @ operator)
mlx_sys::fft           fft (1D), ifft (1D)
mlx_sys::array         construct from raw buffer; expose underlying buffer;
                       eval; reshape; concatenate
```

All bindings are thin wrappers over `mlx::core::array` operations. The Rust API is **graph-building**, not eager: each call returns a handle to a graph node. A separate `eval(roots: &[Handle])` call triggers actual GPU execution. This matches MLX's lazy semantics and is the foundation for whole-subtree fusion.

### MLX subgraph construction model

The fusion entry is `crates/polars-metal-kernels/src/mlx_subgraph.rs` (new). It exposes:

```rust
pub struct MlxSubgraph {
    inputs:  Vec<MlxArrayHandle>,    // per-input-column handle
    outputs: Vec<MlxArrayHandle>,    // per-output-column handle
}

impl MlxSubgraph {
    pub fn from_fusion_scope(
        scope:     &FusionScope,
        column_buffers: &[MetalBuffer],
    ) -> PolarsResult<Self>;

    pub fn eval(&self) -> PolarsResult<Vec<MetalBuffer>>;
}
```

The `from_fusion_scope` constructor:

1. Walks the `FusionScope` (a flattened op list with input refs).
2. Wraps each input column as an `MlxArrayHandle` over the existing `MetalBuffer` (zero-copy: MLX accepts a raw pointer + shape + dtype; the buffer's `MTLBuffer` backing is shared).
3. Emits a chain of MLX graph-building calls — `mlx_sys::sin(handle)`, `mlx_sys::add(a, b)`, etc. — accumulating handles.
4. The terminal op's handle(s) become `outputs`.

`eval()` calls `mlx::core::eval(outputs)` and returns the resulting buffers (zero-copy out: MLX's output `array` shares the same `MTLBuffer` it allocated).

The MLX-side fusion is **automatic** within the subgraph: MLX compiles the constructed graph into one or a small number of Metal compute pipelines and dispatches them. We do not write our own MSL for Phase 8 — the win comes from feeding MLX a whole subgraph instead of bouncing between one-op MLX calls and Polars expression intermediates.

### CPU-parity contract for non-fused ops

Per CLAUDE.md architectural principle 2: every op around the fused subtree must hit CPU parity. M4 makes this measurable.

For each non-fused op that appears in an M4-bench query (Filter, Scan, Select-of-existing-column, HStack-without-compute, materialize-to-Polars-Series), the engine routes to Polars CPU **via zero-copy buffer pass-through**. The check: a benchmark of the non-compute op alone, executed via `engine="metal"`, must come within 5% of the same op executed via `engine="cpu"`. Bench harness in `tests/bench/m4_parity/test_non_compute_ops.py` (new).

This is **not a soft target.** A regression here defeats the M4 thesis even if the compute phase is fast. We enforce it as a perf gate.

## Architecture

### Three-layer flow (unchanged from M3)

Walker (Python) → Router (Rust) → Walker applies plan → Dispatch.

What changes in M4 is what the walker recognizes:

- **M2/M3 walker** identified IR nodes (Filter, GroupBy, etc.) and routed each as a unit.
- **M4 walker** additionally identifies *compute subtrees inside expressions* (not just IR nodes), runs density estimation, marks the subtree as fused, and routes it as a single `MetalPlanNode::FusedExprGraph`.

The IR-node-level routing (Filter, GroupBy, Sort) from M2/M3 is preserved and used for the non-compute glue. The new code is additive.

**Module ownership boundary.** The novel piece is the **fusion analyzer** (`crates/polars-metal-core/src/fusion/analyzer.rs`). It owns: walking the Polars expression IR, identifying maximal compute subtrees, building `FusionScope` per subtree, estimating density. It does **not** own MLX FFI (that lives in `polars-metal-mlx-sys`) or MLX subgraph construction (that lives in `polars-metal-kernels::mlx_subgraph`). The supported-op registry (`supported_ops.rs`) is shared between the analyzer (for op identification) and the subgraph builder (for op→MLX-call mapping).

The MLX FFI surface in `polars-metal-mlx-sys` is the second-largest novelty. It expands from ~1 binding to ~50 over M4. Bindings are kept thin — each is a one-line wrapper over an `mlx::core::*` call. The non-trivial work is the C++↔Rust handle ownership model (using `cxx::SharedPtr<mlx::core::array>`).

The cardinality estimator from M3 (HLL++) is **not used** in M4. M4's density estimator is per-op-static, not data-dependent.

## Algorithm details

### Phase 8.1 — Compute-density estimator

```
Input:  Polars expression IR for one column-producing expression
Output: FusionScope or None

Algorithm:
  Recursive descent over the expression IR (post-order).
  At each node:
    if op is supported (in supported_ops registry) AND all child
      results are supported AND all child results have F32 dtype:
        emit op_id, child handles, FLOPs estimate; bubble up
    else:
        mark this branch as "unsupported boundary"; bubble up
        None for this branch

  Top-level: if root is a "supported" node, we have a maximal subtree.
  Build FusionScope with:
    ops:        post-order list of op-ids
    inputs:     column refs at the unsupported-boundary leaves
    outputs:    one handle (the root)
    est_flops:  sum across ops

  Then density-route decision: if est_flops >= MIN_FLOPS and n_rows
  >= MIN_ROWS, accept the scope. Otherwise, decline.
```

Edge cases:
- Mixed dtypes (F32 chain with one F64 input): cast to F32 if the input is constant-castable; otherwise reject.
- Null bitmaps: the F32 op-set on the GPU side assumes "valid + value" semantics. If any input column has a null bitmap, the analyzer rejects the subtree (Phase 8 deferred; nulls are common but invoke a separate scope-and-route path in a future chunk). Fall back to CPU.
- Multi-rooted subtrees (e.g., `df.with_columns(a=expr1, b=expr2)`): each is a separate scope; fuse if both qualify.

### Phase 8.2 — MLX subgraph build and eval

```
Input:  FusionScope (ops list + input column refs)
        MetalBuffer per input column (zero-copy from Arrow)
Output: MetalBuffer per output column (zero-copy to Arrow)

Algorithm:
  1. For each input column, construct an MlxArrayHandle wrapping the
     MetalBuffer. This is a zero-copy view: MLX accepts a raw pointer
     + shape + dtype, and the underlying MTLBuffer is shared. The
     MlxArrayHandle holds a refcount on the MetalBuffer so it stays
     alive until eval.

  2. Maintain a stack of MlxArrayHandles, indexed by op position.
     Walk the FusionScope ops in post-order:
       case Add:    pop two handles, push mlx_sys::add(a, b)
       case Sin:    pop one handle,  push mlx_sys::sin(a)
       case Where:  pop three (cond, then, else), push mlx_sys::where(...)
       ...
     The handle returned by each MLX call is a graph node (lazy).

  3. The final handle(s) on the stack are the output(s).

  4. Call mlx::core::eval(&[outputs]). MLX compiles the graph into
     one or a small number of Metal compute pipelines and dispatches
     them.

  5. After eval, each output handle backs a freshly-allocated MTLBuffer.
     Construct a MetalBuffer wrapper over it (zero-copy out), then
     wrap as a Polars Series with the buffer-bridge crate's existing
     zero-copy interface.
```

The construction step (1–3) is O(N_ops) and pure CPU work — measured at ~100 µs for a 20-op chain in `benches/mlx_subgraph.rs`. The eval step (4) is the GPU dispatch and is what gets measured against the M4 perf gate. The fold-back (5) is bandwidth-bound and small (< 1 ms for typical output sizes).

### Phase 8.3 — Reductions, sort, and top-k as fused terminuses

Reductions, sort, and top-k terminate a fused subtree differently from element-wise ops: they collapse a column to a scalar (reductions) or reorder it (sort/top-k). The MLX graph builder handles them uniformly — they are just additional op-ids in the FusionScope, with `n_args=1` and a different output shape.

For sort/top-k, the output is a sorted/partitioned column with the same dtype but potentially different shape (top-k cuts to K rows). The fold-back step constructs a fresh Polars Series with the new shape.

For reductions, the output is a scalar (one-row column). The fold-back creates a single-row Polars Series or scalar value depending on the IR context (`pl.col("x").sum()` in an `agg()` vs in a `select()`).

### Phase 9 — Cumsum-diff rolling

Identity: `rolling_mean(x, W)[i] = (cumsum(x)[i] - cumsum(x)[i - W]) / W`, with `cumsum(x)[-1] := 0`. Similarly for rolling_sum.

```rust
pub fn rolling_mean_via_cumsum(
    x:        &MlxArrayHandle,
    window:   usize,
    n_rows:   usize,
) -> MlxArrayHandle {
    let cs = mlx_sys::scan::cumsum(x);
    // Left-shift cs by `window`: prepend `window` zeros, drop last `window`.
    let zeros = mlx_sys::array::zeros(window, F32);
    let cs_head = mlx_sys::array::slice(cs, 0, n_rows - window);
    let cs_shifted = mlx_sys::array::concatenate(&[zeros, cs_head]);
    let diff = mlx_sys::elementwise::sub(cs, cs_shifted);
    mlx_sys::elementwise::div(diff, MlxArrayHandle::scalar(window as f32))
}
```

This is one cumsum + one slice + one concat + one sub + one div. MLX fuses everything except the cumsum and concatenate into one elementwise dispatch. Measured 5.83 ms at N=10M F32 W=1000 (see `bench_rolling_mlx.py`).

For `rolling_var(x, W)`: use the cumsum-of-squares identity:

```
var[i..i+W] = E[x²] - E[x]²
            = (cumsum_x2[i+W] - cumsum_x2[i]) / W
             - ((cumsum_x[i+W] - cumsum_x[i]) / W)²
```

`rolling_std` is `rolling_var.sqrt()`.

Window size must be a compile-time literal at plan time (today's Polars API enforces this). Variable-window rolling (`rolling_*_by`) is out of scope this chunk.

### Phase 10 — List/Array dot-product → MLX matmul

Two source shapes the analyzer recognizes:

1. **`Array[F32, D]`** (fixed-shape array dtype, Polars 1.30+): `df.with_columns(sim=pl.col("emb").arr.dot(query_lit))`.
2. **`List[F32]`** (variable-length list dtype): `df.with_columns(sim=pl.col("emb").list.eval(pl.element() * pl.lit(query)).list.sum())` — the Polars-native columnar matmul. Recognized as long as `pl.element() * pl.lit(...)` is the only inner op and `list.sum()` is the terminus.

In both cases, the analyzer treats the column as an `(N, D)` 2D matrix and the literal as a `(D,)` vector. The MLX dispatch is one matmul:

```
sim = column_matrix @ query_vector       # shape: (N,)
```

Top-K is a separate stage. The analyzer extends to recognize `Sort + Head` or `top_k(K)` on the output and emits an additional `argpartition` op in the subgraph. The end-to-end shape `.with_columns(sim=...).sort("sim", descending=True).head(K)` becomes one fused subgraph (matmul + argpartition + take) that runs in ~5–8 ms at Q=100 N=100k D=768 (measured 4.94 ms in `bench_cosine_topk_mlx.py`).

L2 distance follows the same pattern with the `||q||² + ||c||² - 2q·c` decomposition.

`Array[F32, D]`: M4 prefers this dtype because the underlying buffer is already strided — zero-copy view into MLX is trivial. `List[F32]` is more work because the underlying buffer is variable-stride; the buffer bridge has to verify uniform inner-length before allowing the matmul recognition. If non-uniform, fall back to CPU.

### Phase 11 — `Expr.fft()` API and routing

**New public API.** Today Polars has no `fft` expression. M4 adds one via Polars' expression-extension hook:

```python
@pl.api.register_expr_namespace("metal")
class MetalExprNamespace:
    def fft(self) -> pl.Expr:
        return pl.struct([
            pl.col(...).map_elements(...),   # placeholder; engine intercepts
            ...
        ])
```

The user calls `pl.col("x").metal.fft()`. On Polars CPU, this raises a `NotImplementedError`. On `engine="metal"`, the walker recognizes the `metal.fft` shape and routes through `crates/polars-metal-kernels/src/fft.rs`, which calls `mlx::core::fft::fft` over the input column and returns a struct column `(real F32, imag F32)`.

Alternative considered: `pl.col("x").fft()` directly (no `.metal.` namespace). Rejected because it would require a Polars upstream change. The `.metal.` namespace registration is local and reversible.

The struct output is a Polars `Struct[real: F32, imag: F32]` column. Polars' struct field access (`pl.col("y").struct.field("real")`) lets users get the components.

`Expr.ifft()` follows the same pattern. Multi-dimensional FFT, FFT over List columns, real-input `rfft` (with the conjugate-symmetric optimization) are deferred to Phase 11.next if user demand surfaces.

### Phase 8/9/10/11 — Output fold-back to Polars

All fused-graph executions terminate by handing a `MetalBuffer` back to the Polars buffer-bridge crate. The bridge already does the zero-copy `MTLBuffer` → `Arrow Buffer` wrap (used in M1+). What's new in M4 is recognizing that an MLX output buffer is layout-compatible with Arrow F32 / F64 / Bool / I32 — which it is, by MLX construction.

For the FFT struct output, the fold-back creates two Polars Series (real, imag) sharing the underlying interleaved buffer via stride 2 + offsets 0 and 4.

## Data flow examples

### Example 1: Haversine end-to-end on M2 Ultra

```
df.with_columns(
    dist_km=haversine_expr()    # 4 transcendentals, 6 inputs, 1 output
).collect(engine="metal")
```

```
[Polars IR]
   DataFrameScan
   ↓
   HStack (column: dist_km := haversine(pickup_lat, pickup_lon,
                                          drop_lat,  drop_lon))

[Walker translates IR → MetalPlanNode tree]
   For the HStack:
     - Run fusion analyzer on the haversine expression
     - Subtree is supported: 4 inputs (F32), 4 trig ops, several
       arithmetic ops, output F32
     - est_flops ~ 30 ops × 10M rows = 3e8 FLOPs > 5e7 threshold
     - Accept FusionScope; build MetalPlanNode::FusedExprGraph

[Rust router]
   DataFrameScan: CpuLeave
   HStack:        GpuLift via FusedExprGraph

[Walker applies plan; calls execute_fused_expr]
   1. Wrap 4 input columns as MetalBuffer views (zero-copy from Arrow)
   2. Build MlxSubgraph from FusionScope
   3. mlx::core::eval(outputs)
   4. Wrap output MTLBuffer as MetalBuffer → Polars Series
   5. HStack the result back into the DataFrame (Polars CPU; zero-copy
      column attachment)

[Result returned to user]
```

Expected wall-clock at 10M rows: 3.49 ms MLX eval + ~1 ms graph build + ~1 ms FFI marshalling + ~0.5 ms fold-back ≈ **6 ms** (vs Polars CPU 181 ms = 30× win).

### Example 2: Cosine top-k vector search

```
df = pl.DataFrame({"id": ids, "emb": embeddings})  # emb is Array[F32, 768]
query = np.random.rand(768).astype(np.float32)

result = (
    df.with_columns(sim=pl.col("emb").arr.dot(pl.lit(query)))
      .top_k(10, by="sim")
      .collect(engine="metal")
)
```

```
[Polars IR]
   DataFrameScan
   ↓
   HStack (sim := Array[F32, 768].dot(query_lit))
   ↓
   TopK (k=10, by="sim", descending=True)

[Walker]
   HStack: analyzer sees Array[F32, D].dot(lit). Special-cased
           pattern; emits FusionScope with one op: matmul(emb, lit).
   TopK:   analyzer sees a sort+head on a numeric column produced
           in the previous fused step. Extends the existing scope
           to include an argpartition+take terminus.
   Result: one fused subgraph covering both nodes.

[Rust router]
   DataFrameScan: CpuLeave
   HStack+TopK:   GpuLift via single FusedExprGraph

[Walker applies plan]
   1. Wrap emb column as MlxArrayHandle (view of MTLBuffer over the
      Array's underlying F32 buffer, shape (N, 768))
   2. Wrap query as MlxArrayHandle scalar (shape (768,))
   3. mlx::core::matmul -> sim (shape (N,))
   4. mlx::core::argpartition(-sim, kth=K-1) -> top_idx (shape (N,))
   5. Slice top_idx[:K] -> selected_idx (shape (K,))
   6. mlx::core::take(df, selected_idx, axis=0) -> result (shape (K, ...))
   7. mlx::core::eval
   8. Wrap each output column back as a Polars Series

[Result returned to user]
```

Expected wall-clock at Q=1, N=100k, D=768: ~5 ms (vs Polars-native ~560 ms × Q=1, vs NumPy 1.45 ms × the marshalling = ~6 ms total).

```

[Walker]
   HStack: analyzer sees rolling_mean(window=1000) on F32 column.
           Special-cased pattern. Emits FusionScope as cumsum-diff:
             cs = cumsum(price)

[Walker applies plan]
   1. Wrap price column as MlxArrayHandle
   2. Construct subgraph per the four ops above
   3. mlx::core::eval -> rolling_avg
   4. Wrap back as Polars Series
```

Expected wall-clock at N=10M W=1000: ~7 ms (5.83 ms MLX + 1 ms marshalling).

## Error handling

Carries from M2/M3:

- Unrecognized IR shapes → `Fallback`, ancestors poison, whole query routes to CPU.
- GPU kernel dispatch failures → `polars.exceptions.ComputeError`.
- No partial GPU work within a lifted subtree.

New for M4:

- **Density-threshold rejection** is not an error — it's a routing decision. Subtree falls back to CPU silently; logged at `debug=True`.
- **Unsupported op inside a subtree** → analyzer truncates the scope at the unsupported op boundary; the subtree's surrounding non-fused ops run on CPU. Logged at `debug=True`.
- **Null bitmap on a fused-subtree input column** → analyzer rejects the whole scope; the subtree falls back to CPU. Justification: the Phase 8 op set assumes "valid + value" semantics; null-aware MLX dispatch is deferred.
- **MLX eval failure** (e.g., out-of-memory) → `ComputeError`. We do not silently fall back at execution time.
- **`Expr.fft()` on non-F32 / non-castable column** → `Fallback` at plan time; rolling back to CPU's no-fft means the user gets a `NotImplementedError` from Polars CPU.
- **`Array[F32, D]` with non-uniform inner length** (shouldn't happen by dtype contract, but defensively) → `ComputeError`.
- **`List[F32]` matmul recognition** requires uniform inner length per row. Walker checks at plan time via the column's metadata; if variable, falls back to CPU.

## Testing strategy

Five layers from M2/M3, extended.

### Layer 1: Kernel correctness — Rust proptest

- `test_mlx_subgraph.rs` — synthetic op chains (random combinations of 3–20 supported ops) → eval → byte-equal against pure-Rust scalar reference. Per op-shape × null-density × n_rows bucket: 256 cases. F32 tolerance: bit-exact for arithmetic, ULP-bounded for transcendentals (≤4 ULP).
- `test_rolling_cumsum.rs` — cumsum-diff rolling vs `polars.Series.rolling_mean` ground truth. Edge cases: empty, n=1, n=W (boundary), n<W (all-null prefix), nans.
- `test_list_dot.rs` — matmul correctness vs ndarray reference. Edge cases: D=1, D=2048 (large), N=1, query of all zeros, all ones.
- `test_fft.rs` — MLX FFT vs `numpy.fft.fft`. ULP tolerance documented per N size (FFT roundoff scales as O(log N) per output).

### Layer 1.5: Router & analyzer — Rust unit + proptest

- `test_compute_density.rs` — FLOP-per-op table consistency; threshold behavior at boundaries; arithmetic correctness for total density.

## Risks & open questions (M4-owned)

**Density estimator is wrong.** The static FLOP-per-op table is approximate. If it routes too many subtrees to GPU, we pay dispatch overhead on workloads CPU would have served faster. If too few, we miss wins. *Mitigation:* the threshold is conservative (5e7 FLOPs, ~10M rows × 5 ops). Bench coverage at the threshold boundary verifies the decision. *Open:* if a class of real-world queries clusters near the threshold, the estimator can become data-aware (sample-and-extrapolate) — that's a Phase 12+ refinement.

**MLX kernel-fusion behavior is opaque.** We assume MLX fuses element-wise chains into one Metal kernel; the headline measurements demonstrate it does in practice. *Open:* if MLX's fusion regresses in a future version (it has happened to JAX/PyTorch), our wins would shrink. *Mitigation:* pin MLX version in `Cargo.toml`; bump deliberately. Document the assumed behavior in `docs/architecture.md`.

**FFI marshalling overhead per fold-back.** Going from MLX output → MetalBuffer → Arrow Series is 2 wraps, but each wrap touches refcounts, dtype tags, and possibly metadata. The "CPU parity" target (5% over Polars CPU on non-compute ops) is tight; if FFI marshalling costs more than ~2 ms per query, the parity gate fails. *Mitigation:* benchmark marshalling cost in `benches/mlx_subgraph.rs` as a standalone measurement. *Open:* if marshalling dominates, optimize the buffer-bridge hand-off (already zero-copy in M1+, but the wrap struct construction could batch).

**`Expr.fft()` namespace registration interacts with Polars upstream.** Polars' `register_expr_namespace` API exists but is intended for user namespaces; using it for an engine-specific op feels off-pattern. *Mitigation:* prototype the registration; document the pattern; if it conflicts with Polars upstream, fall back to `pl.col(...).map_batches(metal_fft_func)` (less ergonomic but works without API surface). *Open:* should we upstream a Polars expression `fft` instead? Probably yes long-term; M4 ships the engine-specific path first.

**`List[F32]` non-uniform inner length.** The recognizer needs to verify uniform per-row length to allow matmul routing. This metadata may or may not be cheaply available at plan time. *Mitigation:* if metadata not present, sample-check at plan time (O(N_rows / 1000) reads). *Open:* upstream Polars may eventually expose "uniform-inner-length List" as a typed view, removing the need.

**Null bitmap on F32 input columns.** Phase 8 rejects subtrees with null inputs and falls back to CPU. This is a real coverage gap for production workloads where some columns have nulls. *Mitigation:* document the limitation; benchmark how often it triggers fallback on representative queries. *Open:* null-aware MLX dispatch in a future chunk — either add null-bit masking in the subgraph (cheap per op) or use MLX's `where` to substitute zero for null inputs at reduction terminus (correct semantics for sum / mean but not for std). Substantial design work; deferred from M4.

**Sort/top-k ordering stability.** MLX sort is **not stable** by default. Polars sort is stable. If a user relies on stability, the engine-routed result may differ from CPU. *Mitigation:* the analyzer rejects sort/top-k routing when stability is required (the IR carries a `stable_sort: bool` flag from M3); falls back to CPU. *Open:* implement a stable-sort variant on MLX in a future chunk if user demand surfaces.

**Mixed-dtype subtrees (F32 with one F64 input).** The analyzer casts where it can; falls back when it can't. Cast cost is ~bandwidth-bound and is included in the density estimate. *Open:* edge cases around precision loss aren't tested as heavily as pure-F32 paths. Plan to add a tolerance test in Layer 1 that exercises the F32-after-cast path on inputs that would lose precision.

**MLX FFI surface size.** Adding ~50 bindings is mechanical but bulky. *Mitigation:* generate the bindings from a per-op spec table (one Rust macro that emits a thin wrapper); review each binding's dtype map carefully. *Open:* binding stability across MLX versions — the C++ API of `mlx::core::*` is not API-stable; pin and bump deliberately, same risk as Polars rev.

**Polars rev drift.** Same risk as M1/M2/M3; same mitigation (pin to `py-1.40.1`, bump at milestone boundary). The `Array[F32, D]` dtype was added in Polars 1.30; remain compatible across 1.30+. *Open:* if `pl.col("emb").arr.dot(query)` API changes in Polars 1.41+, M4 will need to update the analyzer pattern-match.
